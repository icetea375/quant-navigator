#### **第一章：战略与心态——“在动手之前，先问对问题”**

**法则一：明确你的“能力边界”，而不是你的“功能列表”。**
*   **要避免的陷阱:** 以“我要实现XX、YY、ZZ功能”来开始一个项目。
*   **正确的开局:** 先诚实地回答三个问题：
    1.  **我有什么“独一无二”的数据？** (e.g., 您的“仲裁判例集”)
    2.  **我有什么“不可复制”的智慧？** (e.g., 您对特定行业的深刻理解)
    3.  **我有什么“无法承受”的成本？** (e.g., 复杂的运维、无尽的标注)
*   **原则:** 你的系统，应该围绕**放大你的“优势”**和**规避你的“劣势”**来构建，而不是去追逐一个大而全的功能列表。

**法则二：接受“AI是强大的实习生，而你是唯一的CEO”这一现实。**
*   **要避免的陷阱:** 试图构建一个“全自动”的、不需要你就能独立思考的“终极AI”。
*   **正确的开局:** 从第一天起，就设计**“人类仲裁”**的机制。将最困难、最模糊、最需要“智慧”的决策点，**明确地**预留给人类。
*   **原则:** AI负责**80%的、重复性的“分析”**，你负责那**20%的、决定性的“判断”**。这个权力边界，神圣不可侵犯。

**法则三：永远选择“商业API”，除非你有“核武器”级别的理由不。**
*   **要避免的陷阱:** “我是一个技术专家，我应该自己微调和部署开源模型。”
*   **正确的开局:** 默认的技术选型**永远是**顶级的、付费的商业LLM API。
*   **原则:** 只有当**同时满足**以下两个条件时，才考虑自建模型：
    1.  你拥有一个**足够大、足够高质量、且竞争对手无法获得的“私有数据集”**（例如，数千条您的仲裁判例）。
    2.  你已经验证，使用通用商业API**确实无法**满足你的性能要求。
*   **记住：** 你的核心竞争力，在于你的**“业务逻辑”和“私有数据”**，而**不是**在于你“部署AI模型”的能力。

#### **第二章：架构与设计——“追求简单，拥抱进化”**

**法则四：你的第一个架构，必须是“健壮的单体”。**
*   **要避免的陷阱:** 一开始就设计一个“可扩展”、“高并发”的微服务架构。
*   **正确的开局:** 将所有后端逻辑，都放在一个**单一的、模块化的**应用中。
*   **原则:** **不要为“一万个用户”时的性能问题，在你只有一个用户（你自己）时，提前支付“复杂性”的代价。** 先用最简单的方式，让系统跑起来。**重构，永远比“预构”便宜。**

**法则五：建立“单一事实来源”，并用机器守护它。**
*   **要避免的陷阱:** 在多个地方，定义同样或类似的东西。
*   **正确的开局:** 为你系统中最重要的“契约”，建立**唯一的、不可辩驳的**“宪法”。
    *   **项目依赖 -> 根 `package.json` / `pyproject.toml`**
    *   **模块路径 -> 主应用的 `tsconfig.json`**
    *   **数据结构 -> `packages/shared-types`**
    *   **架构规则 -> `dependency-cruiser` 配置文件**
*   **原则:** 任何可以被机器**自动化检查和强制执行**的规则，都**绝不能**依赖于人类的“自觉”。

**法则六：面向“接口”而非“实现”编程，为未来“松耦合”。**
*   **要避免的陷阱:** 在业务逻辑中，直接调用一个具体的第三方库或服务（例如，`tushare.get_daily()`）。
*   **正确的开局:** 在你的业务逻辑和外部世界之间，建立一个**抽象的“隔离层”**。
    *   定义一个`DataSourceInterface`，你的业务逻辑只与这个“接口”对话。
    *   然后，创建一个`TushareDataSource`类，去**实现**这个接口。
*   **原则:** 这不是“过度设计”。这是为了确保你系统的“核心业务逻辑”，不会被任何一个具体的、未来可能被替换掉的“外部依赖”所**“绑架”**。

#### **第三章：开发与流程——“手感驱动，测试守护”**

**法则七：你的第一个Sprint，唯一的目标是获得“手感”。**
*   **要避免的陷-阱:** 试图在第一个Sprint就完成所有“后端基础设施”或“前端UI库选型”。
*   **正确的开局:** 用最丑陋、最快速的方式，构建一个**端到端的、能跑通最小业务闭环**的“原型”。
*   **原则:** **在你能“亲手玩到”你的产品之前，你对这个产品的所有假设，都可能是错的。** 必须以最快的速度，进入“反馈-迭代”的循环。

**法则八：将“测试驱动开发(TDD)”作为不可动摇的信仰。**
*   **要避免的陷阱:** “我只是做个小修改，先不写测试了。”
*   **正确的开局:**
    1.  **先写一个会失败的测试，来“定义”你要做什么。**
    2.  **再写代码，来“完成”这个定义。**
*   **原则:** 测试，不是“代码的附属品”，而是**“代码的驱动力”**和**“设计的规格书”**。

**法则九：用“决策日志(ADR)”来对抗“遗忘”。**
*   **要避免的陷-阱:** 相信自己能永远记住当初为什么做出某个重要的技术决策。
*   **正确的开局:** 每当你做出一个**重要的、有权衡的、非显而易见的**决策时，花5分钟，在`ADR.md`中，记录下你的**“思考过程”**。
*   **原则:** 这份文档，是写给**“未来的、已经忘记了一切的你自己”**看的。它是你对抗项目长期熵增和认知衰退的唯一武器。

#### **法则七（新增）：【专业分工铁律】尊重“生态边界”，禁止“语言乌托邦”**

*   **核心原则:** **用最适合的工具，去做最适合的事 (Use the right tool for the right job)。** 承认并尊重不同编程语言及其生态系统，是在不同领域（如后端计算 vs 前端交互）经过数十年“自然选择”后形成的专业优势。

*   **要避免的陷阱:** 为了追求所谓“开发语言的统一”，而强行使用一种语言，去“跨界”完成它本身并不擅长的工作。例如，使用Python去构建复杂的前端用户界面，或使用Node.js去进行大规模的科学计算。

*   **正确的开局:** 在项目启动时，为系统中**不同性质的“领域”**，选择**该领域内最强大、生态最成熟**的技术栈，即使这意味着引入多种语言。
    *   **后端 (数据处理、AI/ML、核心业务逻辑):** **永远首选 Python。** 这是“计算”和“逻辑”的世界，Python拥有无可匹敌的生态系统。
    *   **前端 (用户界面、交互体验):** **永远首选 TypeScript + (React/Vue)。** 这是“展示”和“交互”的世界，JavaScript/TypeScript生态是唯一的成熟选择。
    *   **移动端 (如果需要):** 选择 Swift/Kotlin (原生) 或 React Native/Flutter (跨平台)。

*   **架构师的真正职责:**
    *   架构师的职责，**不是**消灭语言的多样性，而是**管理**这种多样性所带来的**“边界”**。
    *   **【关键行动指令】** 对于系统中不同语言的“独立王国”，必须建立**两条不可动摇的“边境管理”机制**：
        1.  **“自动化的大使馆 (Automated Embassy)”:**
            *   **机制:** 必须建立**自动化的“契约生成与共享”流水线**（例如，使用OpenAPI规范，从Python/FastAPI后端，自动生成TypeScript的前端类型定义）。
            *   **目的:** 用**机器**来保证“数据契约”的**语法层面**100%同步，根除前后端因数据结构不匹配而产生的Bug。
        2.  **“标准化的外交照会 (Standardized Diplomatic Note)”:**
            *   **机制:** 必须建立**“API变更日志 (API_CHANGELOG.md)”**制度。
            *   **目的:** 用**文档和流程**来保证，任何破坏性的API变更，其背后的**“意图 (Why)”**，都能在**人类层面**被清晰地沟通和理解。

*   **最终的“反脆弱”设计:**
    *   一个由“Python后端”和“TypeScript前端”组成的系统，其“脆弱点”只存在于它们之间那条**细细的“API边界”**上。
    *   通过用“自动化契约生成”和“变更日志”这两把“金锁”，将这条边界**焊死**，我们就构建了一个**“内部各自为王、外部邦交稳固”**的反脆弱系统。
    *   这个系统，既能最大化地利用每一个领域**最顶级的生态优势**，又能将跨语言协作的**风险和成本，降至最低**。
