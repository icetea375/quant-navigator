### 数据库架构说明

#### 极简主义数据库设计
项目采用**PostgreSQL + Redis + 本地文件系统**的简化架构：

```sql
-- 数据库: papa_quant
-- 包含以下核心表：

-- 1. 核心业务表
- human_feedback_loop         -- 人工仲裁案例和反馈循环
- system_configs             -- 系统配置参数
- analysis_results           -- 主工作流分析结果

-- 2. 存储策略
- PostgreSQL: 结构化业务数据、系统配置、人工反馈、分析结果
- Redis: 缓存和会话管理
- 本地文件系统: Prompt模板、知识库文件、仲裁案例
```

#### 数据库优势
1. **极简架构**：只保留核心业务表，简化数据管理
2. **成本优化**：减少数据库复杂性和维护成本
3. **性能提升**：避免复杂的跨表查询和关联
4. **易于维护**：清晰的数据结构和职责分离
5. **快速部署**：简化的数据库配置和迁移
6. **灵活扩展**：核心资产存储在文件系统中，便于版本控制

### 开发规范更新

#### 1. 文件命名规范
- **Python脚本**: 使用snake_case，如 `main_workflow.py`
- **配置文件**: 使用kebab-case，如 `attribution-rules.json`
- **Prompt模板**: 使用描述性名称，如 `attribution_analysis_prompt.txt`
- **模块文件**: 使用camelCase，如 `admin.module.ts`

#### 2. 目录组织规范
- **按功能分组**: 相关功能放在同一目录下
- **按层次分组**: 主工作流、支持模块、核心资产分层组织
- **按环境分组**: 开发、测试、生产环境分离
- **按模块分组**: 主应用、AI治理中心、核心资产分离

#### 3. 配置管理规范
- **环境变量**: 使用 `.env` 文件管理配置
- **配置文件**: 使用 JSON 格式存储复杂配置
- **数据库配置**: 使用 PostgreSQL + Redis + 本地文件系统

```python
    # 6. 最终预测生成
    prediction = generate_prediction(event_chain)

    # 7. 反事实验证
    counterfactual = verify_counterfactual(prediction)

    # 8. 人工仲裁
    if needs_human_review(prediction):
        human_arbitrator.collect_feedback(prediction)

    return prediction
```

- **Python脚本**: 使用snake_case，如 `main_workflow.py`
- **配置文件**: 使用kebab-case，如 `attribution-rules.json`
- **Prompt模板**: 使用描述性名称，如 `attribution_analysis_prompt.txt`
- **模块文件**: 使用camelCase，如 `admin.module.ts`

### 2. 目录组织规范

- **按功能分组**: 相关功能放在同一目录下
- **按层次分组**: 主工作流、支持模块、核心资产分层组织
- **按环境分组**: 开发、测试、生产环境分离
- **按模块分组**: 主应用、AI治理中心、核心资产分离

### 3. 配置管理规范 (v11.4 中心化配置革命)

**配置架构原则**:
- **单一事实来源**: 所有配置通过 `config/settings.py` 统一管理
- **确定性优先级**: 代码定义的配置加载顺序
- **类型安全**: 强类型配置接口
- **环境隔离**: 开发/测试/生产环境完全分离
