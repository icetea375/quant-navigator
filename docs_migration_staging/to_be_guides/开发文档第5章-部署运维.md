# 第5章：测试与质量保证 (The Constitution)

## 5.1 测试宪法 (v13.0 TDD最终版)

### 5.1.1 测试宪法概述

**测试宪法 (`v13.0`)**: 本章即为项目的"测试宪法"，是不可动摇的最高准则。

**核心原则**:
1. **TDD驱动开发 (第0条)**: 必须遵循"测试驱动开发 (Test-Driven Development)"
2. **类型安全铁律 (第4条)**: 严禁`as any`, `@ts-ignore`
3. **环境一致性铁律 (第9条)**: 测试必须运行在与生产同构的环境中（通过`@nestjs/testing`和Docker）
4. **CI/CD最终仲裁 (第11条)**: GitHub Actions是最终的、不可逾越的"守门人"

### 5.1.2 测试分层架构

#### 5.1.2.1 单元测试 (Unit Tests)
- **覆盖率要求**: 90% 以上代码覆盖率
- **执行时间**: < 1秒 (单个测试)
- **依赖**: 无外部依赖，使用Mock
- **工具**: Jest (JavaScript/TypeScript), pytest (Python)

#### 5.1.2.2 集成测试 (Integration Tests)
- **覆盖率要求**: 80% 以上关键路径覆盖率
- **执行时间**: < 10秒 (单个测试)
- **依赖**: 测试数据库，模拟外部服务
- **工具**: Jest + Supertest, pytest + TestClient

#### 5.1.2.3 端到端测试 (E2E Tests)
- **覆盖率要求**: 100% 核心用户场景覆盖率
- **执行时间**: < 30秒 (单个测试)
- **依赖**: 完整测试环境
- **工具**: Playwright (Web), Cypress (Web)

### 5.1.3 测试质量标准

#### 5.1.3.1 测试编写标准
- **命名规范**: `should [expected behavior] when [condition]`
- **AAA模式**: Arrange-Act-Assert
- **独立性**: 每个测试独立运行，无共享状态
- **可读性**: 测试代码应该像文档一样清晰

#### 5.1.3.2 测试维护标准
- **及时更新**: 代码变更时同步更新测试
- **定期审查**: 每月审查测试质量和覆盖率
- **重构优化**: 定期重构测试代码，提高可维护性
