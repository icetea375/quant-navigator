# ADR-005: 采用模块化单体架构模式

**状态**: 已接受  
**日期**: 2025年1月17日  
**决策者**: 技术团队  
**影响范围**: 整体系统架构

## 背景

智能分析系统需要处理复杂的金融数据分析、AI治理和量化信号生成等功能。在架构选择时，我们面临以下挑战：

1. **功能复杂性**: 系统包含多个相关但相对独立的功能模块
2. **开发效率**: 需要支持多个团队并行开发
3. **部署简化**: 希望简化部署和运维复杂度
4. **未来扩展**: 需要考虑未来可能的微服务拆分需求

## 决策

选择**模块化单体 (Modular Monolith)** 架构模式作为系统的主要架构风格。

## 理由

### 1. 开发效率优势
- **统一技术栈**: 所有模块使用相同的技术栈，降低学习成本
- **代码共享**: 便于共享通用组件、工具函数和类型定义
- **原子提交**: 跨模块的变更可以在一个提交中完成
- **简化调试**: 所有代码在同一个进程中，便于调试和问题排查

### 2. 部署运维简化
- **单一部署单元**: 整个应用作为一个单元进行部署
- **简化监控**: 只需要监控一个应用实例
- **降低复杂度**: 避免微服务的网络调用、服务发现等复杂性

### 3. 性能优势
- **进程内调用**: 模块间通信通过函数调用，性能更高
- **共享内存**: 模块间可以共享数据，减少重复计算
- **无网络延迟**: 避免微服务间的网络调用延迟

### 4. 未来可扩展性
- **模块边界清晰**: 每个模块都有明确的接口和职责
- **易于拆分**: 当需要时，模块可以轻松拆分为独立的微服务
- **渐进式演进**: 可以根据业务需要逐步演进架构

## 替代方案

### 1. 微服务架构
**优点**:
- 技术栈多样性
- 独立部署和扩展
- 团队自治

**缺点**:
- 运维复杂度高
- 网络调用延迟
- 数据一致性复杂
- 调试困难

### 2. 传统单体架构
**优点**:
- 简单直接
- 部署简单

**缺点**:
- 模块耦合度高
- 难以并行开发
- 扩展性差

## 后果

### 正面影响
- **开发效率提升**: 统一的技术栈和工具链
- **部署简化**: 单一部署单元，降低运维复杂度
- **性能优秀**: 进程内调用，无网络延迟
- **维护成本低**: 相对简单的架构，易于理解和维护

### 负面影响
- **技术栈限制**: 所有模块必须使用相同的技术栈
- **团队耦合**: 模块间存在一定的依赖关系
- **扩展限制**: 无法独立扩展单个模块

### 风险缓解
- **模块边界设计**: 通过清晰的接口设计降低模块间耦合
- **代码组织**: 使用Monorepo管理代码，保持模块独立性
- **文档规范**: 建立完善的文档体系，记录模块间的关系和依赖
- **测试策略**: 通过单元测试和集成测试保证模块质量

## 后果 (Consequences)

### 正面影响 (已实现)
- **开发效率提升**: 统一的技术栈确实提高了开发效率
- **部署简化**: 单一部署单元降低了运维复杂度
- **性能优秀**: 进程内调用避免了网络延迟
- **代码共享**: Monorepo结构促进了代码共享

### 负面影响 (已发现)
- **架构混乱**: 存在两个后端服务 (`backend/` 和 `backend-python/`)，造成架构不清晰
  - **根本原因**: 项目演进过程中，从Node.js混合架构向纯Python架构迁移
  - **具体问题**: `backend/`包含Python+Node.js混合代码，`backend-python/`是新的纯Python实现
  - **影响**: 新开发者不知道应该使用哪个后端，代码重复，维护成本翻倍
- **技术债务**: 混合架构增加了维护复杂度
- **团队困惑**: 新开发者难以理解为什么有两个后端
- **测试复杂**: 需要维护两套测试体系

### 意外后果 (未预料到)
- **文档不一致**: 文档描述与实际架构不符
- **工具链复杂**: 需要同时支持Python和Node.js工具链
- **依赖管理**: pnpm和pip的依赖管理存在冲突

### 经验教训
1. **架构演进需要更清晰的迁移计划**
2. **文档必须与实际实现保持同步**
3. **技术债务的积累速度比预期更快**
4. **需要更严格的架构审查流程**

## 实施细节

### 1. 模块划分原则
- **按业务功能划分**: 每个模块负责一个明确的业务领域
- **接口优先**: 模块间通过明确定义的接口进行通信
- **数据隔离**: 每个模块管理自己的数据模型
- **独立测试**: 每个模块可以独立进行单元测试

### 2. 技术实现
- **Monorepo结构**: 使用pnpm管理多个包
- **共享类型**: 通过shared-types包共享类型定义
- **API层**: 使用FastAPI提供统一的API接口
- **前端分离**: 前端应用独立开发和部署

### 3. 实际模块结构
```
papa/
├── packages/
│   ├── backend/            # 主后端服务 (Python + Node.js混合)
│   │   ├── src/            # Python核心业务逻辑
│   │   ├── support_modules/ # Python支持模块
│   │   └── tests/          # 后端测试
│   ├── backend-python/     # 纯Python后端服务 (新架构)
│   │   ├── src/            # Python服务层
│   │   └── tests/          # Python测试
│   ├── frontend-main/      # 主前端应用 (Vue.js)
│   ├── frontend-aigc/      # AIGC前端应用 (React)
│   └── shared-types/       # 共享类型定义
├── tools/
│   └── tests/              # 统一测试套件
└── docs/                   # 项目文档
```

**注意**: 当前存在两个后端服务，这是架构演进过程中的过渡状态。
