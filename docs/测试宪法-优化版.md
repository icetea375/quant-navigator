### **【v1t0.12 测试宪法】—— 智能分析系统测试实施与验收指南**

**前言：** 本指南是"量化导航仪"项目所有测试开发的最高行为准则。所有提交的测试代码，无论由人类还是AI编写，都必须100%遵循此指南。任何违反本指南原则的代码提交，都将被视为**不合格**并被拒绝合并。

**核心哲学："一个人的军队"原则**

本项目遵循"一个人的军队"开发哲学，将**简单性**和**可维护性**作为最高优先级。我们拒绝"过度工程化"，追求"足够好"的解决方案。测试策略必须与这一哲学保持一致：避免引入不必要的复杂性，优先选择简单、直接、易于理解和维护的测试方法。

---

#### **第一章：核心哲学——"测试即契约"**

**第1条：【测试的唯一目的】**
测试的唯一目的，是验证"生产代码"是否严格履行了其**"面向用户的、可观察的行为契约"**。测试**不应该**关心、也**不应该**断言任何关于代码**"内部实现细节"**的东西。

*   **合宪的行为：** `expect(screen.getByText('加载中...')).toBeInTheDocument();` (测试用户看到了什么)
*   **违宪的行为：** `expect(wrapper.vm.isLoading).toBe(true);` (测试内部状态，严重违宪)

**第2条：【"红灯-绿灯-重构"是唯一的开发节奏】**
所有新功能的开发、所有Bug的修复，都**必须**遵循T-D-D（测试驱动开发）的"红-绿-重构"循环。**禁止**编写任何一行没有先行为其编写一个"失败的测试"的生产代码。

**第3条：【禁止"为了通过而欺骗"】**
任何为了让测试"变绿"而采取的、削弱测试有效性的"欺骗"行为，都将被视为最高级别的叛国罪。这包括但不限于：
*   在生产代码中，添加`if (process.env.NODE_ENV === 'test')`之类的逻辑分支。
*   将被测试的行为本身注释掉。
*   将一个有意义的断言，修改为一个无意义的、恒为真的断言。

**第4条：【"简单性优先"铁律】**
*   **拒绝过度工程化：** 严禁为了追求理论上的"完美"而引入不必要的复杂性。
*   **选择"足够好"的解决方案：** 当面临多种测试策略选择时，优先选择最简单、最直接、最容易理解和维护的方案。
*   **成本效益原则：** 测试方案的成本必须与其带来的收益相匹配。避免"用大炮打蚊子"。

---

#### **第二章：代码级实施铁律**

**第5条：【类型安全是绝对的】**
在所有`.test.ts`文件中：
*   **绝对禁止**使用`as any`、`@ts-ignore`或`@ts-nocheck`。
*   **如果出现类型错误，唯一的、100%的解决方案是：** 回到源头（生产代码、类型定义、测试数据），去修复那个导致类型不匹配的、**错误的根源**。

**第6条：【模拟的边界是神圣的】**
*   **只允许模拟"跨越网络边界"的外部依赖：**
    1.  API Service层（例如`arbitrationService.ts`）。
    2.  最底层的浏览器API（例如`window.matchMedia`）。
*   **绝对禁止模拟：**
    1.  Pinia Store、Vue Router、Element Plus组件。
    2.  任何我们自己编写的、属于本项目一部分的子组件或组合式函数。
*   **原则：** 测试必须在一个**尽可能接近"真实"**的、包含了**完整插件生态**的环境中运行。

**第7条：【断言必须面向"用户"】**
*   **优先使用`@testing-library/vue`的`screen`查询。** 以**真实用户（包括屏幕阅读器用户）**的视角查找和断言元素。
*   **查询的优先级顺序（不可违背）：**
    1.  `getByRole`: 按"可访问性"角色查找（最健壮）。
    2.  `getByLabelText`: 按表单标签查找。
    3.  `getByText`: 按元素显示的文本内容查找。
    4.  `getByTestId`: 作为**最后的、万不得已的**手段。
*   **绝对禁止**使用基于CSS类名或HTML标签的、脆弱的选择器。

**第8条：【断言必须"精确且有意义"】**
*   **禁止"存在性"断言:** 避免使用 `expect(result).not.toBeNull()` 或 `expect(result).toBeDefined()` 这种几乎没有信息量的断言。
*   **必须使用"值"断言:** 断言必须检查**具体的、预期的**值。
    *   **好的实践:** `expect(result.status_code).toBe(200);`
    *   **好的实践:** `expect(result.data.length).toBeGreaterThan(0);`

---

#### **第三章：前端测试专门宪法**

**第9条：【`createTestWrapper`是唯一的创世函数】**
本项目`frontend-main`包中，所有Vue组件的挂载，**【必须】**且**【只能】**通过调用位于`/src/utils/test-utils.ts`中的`createTestWrapper`函数来完成。任何直接调用`@vue/test-utils`的`mount`或`@testing-library/vue`的`render`的行为，都将被视为严重的违宪行为。

**第10条：【真实浏览器是唯一的战场】**
*   所有组件和集成测试，都**必须**运行在由`@vitest/browser`所提供的**真实浏览器环境**中（当前为Chromium）。
*   **彻底废除**并**禁止**使用任何基于`JSDOM`或`happy-dom`的模拟DOM环境。我们接受真实浏览器环境带来的"慢"，以换取其无可替代的"真"。

---

#### **第四章：自动化执行与强制约束**

**第11条：【所有测试必须在提交前通过】**
*   在 `package.json` 的 `scripts` 中，配置一个 `pre-commit` 或 `pre-push` 的 **Git钩子 (Git Hook)**。
*   **指令:** `npm run test`。
*   **效果:** 任何开发者（包括AI）在尝试提交代码时，本地的所有测试都会被**自动触发**。只要有一个测试失败，**`git commit` 或 `git push` 命令就会被自动阻止**。

**第12条：【CI/CD是最终的、唯一的、无情的审判官】**
*   一个功能分支，只有在**同时满足**以下所有由CI/CD流水线**自动强制执行**的条件时，才**允许**被合并到主分支：
    1.  `pnpm run lint` **0个错误**。
    2.  `pnpm run type-check` **0个错误**。
    3.  `pnpm run test:coverage` **100%测试用例通过**。
    4.  由`vitest`生成的覆盖率报告，其**行覆盖率 (Lines) 不得低于85%**，**分支覆盖率 (Branches) 不得低于80%**。
*   **任何试图绕过、禁用或降低这些门禁标准**的行为，都将被视为对本项目根基的攻击。

**第13条：【环境一致性铁律】**
*   **虚拟环境激活:** 对于Python项目，**必须**在运行测试前激活正确的虚拟环境。
*   **依赖一致:** 测试环境**必须**通过 `pnpm install`，安装与 `package.json` 中**完全一致**的生产和开发依赖。
*   **框架一致:** 对于后端NestJS测试，**必须**使用 `@nestjs/testing` 模块提供的 `Test.createTestingModule()` 来构建测试模块。

---

#### **第五章：测试金字塔与分层策略**

**第14条：【单元测试验收标准】**
*   每一个独立的**公共方法 (public method)**，都必须有对应的单元测试。
*   每一个**数据校验规则 (`class-validator`)**，都必须有至少一个**成功**和一个**失败**的测试用例。
*   代码覆盖率的**门禁**: **行覆盖率不得低于85%，分支覆盖率不得低于80%。** 任何低于此标准的提交将被自动拒绝。

**第15条：【集成测试验收标准】**
*   每一个**API端点 (Endpoint)**，都必须有集成测试。
*   测试用例必须覆盖**"成功路径 (Happy Path)"**、**"认证/授权失败"**和**"输入数据校验失败"**这三种核心场景。
*   对于涉及数据库写入的测试，**必须**在测试结束后，通过 `afterEach` 或 `afterAll` 钩子，**清理掉**本次测试创建的所有数据，确保测试的独立性和可重复性。

**第16条：【端到端测试验收标准】**
*   必须覆盖**至少一条**最核心的、贯穿整个系统的用户工作流。
*   E2E测试运行在与生产环境**几乎完全一样**的独立环境中，包含真实的测试数据库和真实的API密钥。
*   E2E测试在CI/CD中设置为**"非阻塞 (Non-blocking)"**模式，失败时发送告警，由人类专家判断是"真实世界的暂时性抖动"还是"真正的代码bug"。

---

#### **第六章：版本控制与工作流**

**第17条：【原子化提交铁律】**
*   **定义：** 每一次 `git commit` 都必须代表一个**单一的、逻辑上完整的**工作单元。
*   **提交的完整性：** 生产代码和测试代码必须在一个提交中。它们是同一枚硬币的两面，绝对禁止分离。
*   **实践准则：** 在执行 `git commit` 前，先使用 `git status` 和 `git diff --staged` 检查暂存区，确保只有与本次逻辑单元相关的代码被提交。

**第18条：【提交信息即文档铁律】**
*   **规范：** 所有提交信息**必须**遵循"**[Conventional Commits](https://www.conventionalcommits.org/)**"规范。
*   **格式：** `<类型>(<范围>): <主题>`
    *   **类型 (Type):** `feat`, `fix`, `refactor`, **`test`**, `docs`, `chore`
    *   **范围 (Scope):** 可选，指明本次提交影响的模块
    *   **主题 (Subject):** 简明扼要地描述变更内容

**第19条：【主干常绿，分支工作铁律】**
*   **主干保护：** `main` 分支是神圣的。任何时候，`main` 分支的代码都必须是**100%通过所有测试**的、随时可以部署到生产环境的状态。
*   **特性分支工作流：** 所有新的开发工作都**必须**从最新的 `main` 分支创建一条新的**特性分支**，通过**拉取请求**申请合并回 `main` 分支。
*   **合并的唯一前提：** 一个Pull Request**只允许**在通过了所有CI/CD流水线检查之后，才能被合并。

---

#### **第七章：文档即基石**

**第20条：【文档的"单一事实来源"铁律】**
*   本项目所有的人类可读的、持久化的技术文档，其**唯一的、排他的**事实来源，**必须**是位于项目根目录下的 `/docs` 目录。
*   文档格式**必须**为 Markdown (`.md`)。
*   **绝对禁止**任何形式的"自建文档"，包括但不限于本地的Word/Pages文档、备忘录、离线的个人笔记等。

**第21条：【代码的"自文档"铁律】**
*   所有**公开的 (public)** 模块、类、函数和方法，都**必须**拥有符合 **Google Python Style Guide** 规范的**文档字符串 (Docstring)**。
*   **Docstring的核心任务**是解释：意图和职责、参数、返回值、可能抛出的异常、使用示例。

**第22条：【文档与开发同步铁律】**
*   文档的更新，**不是**开发和测试完成后的一个独立步骤，而是它们的**一部分**。
*   **"完成的定义"**：一个功能、修复或重构，只有在以下所有条件都满足时，才能被认为是"完成的"：
    1.  生产代码已实现。
    2.  单元/集成测试已通过。
    3.  **所有相关的Docstring和`/docs`目录下的架构文档，都已同步更新。**

**总结：**

通过建立"测试即契约"的宪法原则，我们构建了项目的**三大支柱**：

1.  **代码 (Code):** 负责实现 - "如何做"
2.  **测试 (Tests):** 负责验证 - "做了什么"
3.  **文档 (Docs):** 负责解释 - "为何如此"

这三者共同构成了一个**健康的、可维护的、能够抵抗时间侵蚀的**项目生态系统，有效对抗"知识的熵增定律"，防止"项目失忆症"的发生。
