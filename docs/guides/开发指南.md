## 开发流程

### 1. 测试驱动开发 (TDD)

#### 1.1 TDD铁律
- **核心原则**: 先写测试，再写代码
- **开发循环**: 红-绿-重构 (Red-Green-Refactor)
- **验收标准**: 任何没有对应测试的生产代码提交都将被拒绝

#### 1.2 TDD流程
```bash
# 1. 红灯阶段 - 编写失败的测试
npm run test:unit

# 2. 绿灯阶段 - 编写最简代码让测试通过
# 编写生产代码

# 3. 重构阶段 - 优化代码结构
npm run lint:fix
npm run test:unit
```

### 2. 代码质量保证

#### 2.1 代码规范
- **Python**: 遵循PEP 8，使用Ruff进行代码检查
- **TypeScript**: 遵循ESLint配置，使用Prettier格式化
- **文档**: 所有公共API必须有完整的docstring

#### 2.2 质量门禁
```bash
# 代码检查
npm run lint

# 类型检查
npm run type-check

# 测试执行
npm run test

# 文档质量检查
npm run docs:check
```

### 3. 文档要求

#### 3.1 代码文档
- **Docstring覆盖率**: ≥80%
- **格式标准**: Google风格docstring
- **必需部分**: Args, Returns, Raises
- **推荐部分**: Examples, 详细描述

#### 3.2 架构文档
- **ADR记录**: 所有重要决策必须记录在ADR中
- **设计文档**: 模块设计必须有对应的设计文档
- **API文档**: 所有API必须有完整的接口文档

## 分支管理

### 1. 分支策略

#### 1.1 分支类型
- **main**: 主分支，永远可部署
- **develop**: 开发分支，集成最新功能
- **feature/**: 功能分支，开发新功能
- **fix/**: 修复分支，修复bug
- **hotfix/**: 热修复分支，紧急修复

#### 1.2 分支命名规范
```bash
# 功能分支
feature/add-user-authentication
feature/quant-signal-engine

# 修复分支
fix/data-pipeline-nan-error
fix/api-response-format

# 热修复分支
hotfix/security-vulnerability
```

### 2. 提交规范

#### 2.1 提交信息格式
```
<类型>(<范围>): <主题>

<详细描述>

<相关issue>
```

#### 2.2 提交类型
- **feat**: 新功能
- **fix**: Bug修复
- **docs**: 文档更新
- **style**: 代码格式调整
- **refactor**: 代码重构
- **test**: 测试相关
- **chore**: 构建流程、辅助工具等

#### 2.3 提交示例
```bash
feat(QuantSignal): 添加异常检测算法

实现了基于Z-Score的异常检测算法，支持：
- 价格异常检测
- 成交量异常检测
- 信号强度计算

Closes #123
```

## 测试策略

### 1. 测试分类

#### 1.1 单元测试
- **覆盖率要求**: ≥85%
- **测试范围**: 所有公共函数和类
- **执行命令**: `npm run test:unit`

#### 1.2 集成测试
- **测试范围**: API端点、数据库操作
- **数据隔离**: 每个测试独立的数据环境
- **执行命令**: `npm run test:integration`

#### 1.3 端到端测试
- **测试范围**: 完整用户工作流
- **环境要求**: 与生产环境一致
- **执行命令**: `npm run test:e2e`

### 2. 测试编写规范

#### 2.1 测试命名
```python
def test_calculate_quant_signal_with_valid_data():
    """测试使用有效数据计算量化信号"""
    pass

def test_calculate_quant_signal_raises_error_with_invalid_data():
    """测试使用无效数据时抛出异常"""
    pass
```

#### 2.2 测试结构
```python
def test_function_name():
    """测试函数功能描述"""
    # Arrange - 准备测试数据
    input_data = create_test_data()

    # Act - 执行被测试的功能
    result = function_under_test(input_data)

    # Assert - 验证结果
    assert result.status == "success"
    assert result.data is not None
```

## 部署流程

### 1. 环境配置

#### 1.1 环境变量
```bash
# 开发环境
export NODE_ENV=development
export DATABASE_URL=postgresql://postgres:password@localhost:5432/quant_navigator_dev
export REDIS_URL=redis://localhost:6379

# 测试环境
export NODE_ENV=test
export DATABASE_URL=postgresql://postgres:testpass@localhost:5432/quant_navigator_test
export REDIS_URL=redis://localhost:6379/1

# 生产环境
export NODE_ENV=production
export DATABASE_URL=postgresql://user:pass@prod-db:5432/quant_navigator
export REDIS_URL=redis://prod-redis:6379
```

#### 1.2 配置文件
- **开发环境**: `config/env/env.development`
- **测试环境**: `config/env/env.test`
- **生产环境**: `config/env/env.production`

### 2. 部署步骤

#### 2.1 开发环境部署
```bash
# 1. 启动数据库
docker-compose up postgres redis -d

# 2. 运行数据库迁移
python scripts/migrate.py

# 3. 启动后端服务
cd packages/backend-python
python main.py

# 4. 启动前端服务
cd packages/frontend-main
npm run dev
```

#### 2.2 生产环境部署
```bash
# 1. 构建应用
npm run build

# 2. 运行测试
npm run test

# 3. 部署到服务器
docker-compose -f docker-compose.prod.yml up -d
```

## 最佳实践

### 1. 代码编写

#### 1.1 函数设计
- **单一职责**: 每个函数只做一件事
- **参数限制**: 函数参数不超过5个
- **返回值明确**: 返回值类型和含义要清晰
- **异常处理**: 合理处理异常情况

#### 1.2 类设计
- **封装性**: 合理使用私有属性和方法
- **继承层次**: 避免过深的继承层次
- **接口设计**: 定义清晰的公共接口
- **文档完整**: 类和方法都有完整的文档

### 2. 测试编写

#### 2.1 测试原则
- **独立性**: 测试之间相互独立
- **可重复**: 测试结果可重复
- **快速执行**: 单元测试执行时间短
- **覆盖全面**: 覆盖所有重要路径

#### 2.2 测试数据
- **测试Fixtures**: 使用可重用的测试数据
- **Mock对象**: 合理使用Mock减少依赖
- **数据清理**: 测试后清理测试数据

### 3. 文档维护

#### 3.1 文档更新
- **同步更新**: 代码变更时同步更新文档
- **版本控制**: 文档与代码使用相同的版本控制
- **质量检查**: 定期检查文档质量

#### 3.2 知识传承
- **设计决策**: 重要决策记录在ADR中
- **实现细节**: 复杂逻辑在代码注释中说明
- **业务背景**: 业务规则在文档中详细说明
