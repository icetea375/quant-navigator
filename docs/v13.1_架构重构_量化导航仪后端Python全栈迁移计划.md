# 【v13.1 架构重构】"量化导航仪"后端Python全栈迁移计划

**核心目标：** 将所有后端服务（包括`AI治理中心`的API）统一到Python技术栈，由`FastAPI`框架承载，同时保留并增强我们已经确立的所有先进设计（TDD、模块化、人机协同等）。

**总计预计耗时: 1 - 2 周** (这笔投资，将在未来为您节省数月的时间)

---

## 第一阶段：环境准备与"新宪法"的确立 (1-2天)

**目标:** 搭建一个统一的、纯Python的后端开发与测试环境。

### 1. 【代码库重构】创建新的目录结构

*   在您的papa根目录下，创建一个新的包/目录，例如 `packages/backend-python/`。
*   将您现有的所有Python脚本（`main_workflow.py`, `data_pipeline.py`等）和`config/`目录，全部**移动**到这个新目录中。
*   **归档旧的后端:** 将`backend/` (NestJS) 和 `aigc/backend/` (NestJS) 两个目录，重命名为 `_archive_backend_nestjs/`，暂时保留以备参考，但它们将不再是项目的一部分。

### 2. 【依赖管理】建立唯一的 `pyproject.toml`

*   在 `packages/backend-python/` 目录下，使用 `Poetry` (强烈推荐) 或`pipenv`，初始化一个新的Python项目。
*   创建一个**唯一的 `pyproject.toml` 文件**，它将管理**所有**后端相关的依赖，包括：
    *   **Web框架:** `fastapi`, `uvicorn`, `pydantic`
    *   **数据科学:** `pandas`, `numpy`, `lightgbm`
    *   **AI & API:** `openai`, `zhipuai`, `dashscope` 等商业LLM SDK
    *   **数据库:** `sqlalchemy`, `psycopg2-binary`, `alembic` (用于数据库迁移)
    *   **测试:** `pytest`, `pytest-cov`, `httpx` (用于API集成测试)

### 3. 【测试"宪法"修订】统一到 `pytest`

*   删除所有与 `Jest` 相关的配置文件 (`jest.config.js`等)。
*   在 `packages/backend-python/` 目录下，创建 `pytest.ini` 和 `tests/` 目录。
*   **`conftest.py` 将成为测试的"心脏"**，负责提供数据库连接、模拟对象等所有测试夹具 (fixtures)。

## 第二阶段："API层"的TDD式重建 (2-4天)

**目标:** 使用TDD原则，将之前在NestJS中实现的所有API功能，用`FastAPI`进行1:1的重建。

### 工作流 (严格遵循TDD)

1.  **【红灯】为"AI治理中心"的API编写测试:**
    *   在 `packages/backend-python/tests/api/` 目录下，创建一个 `test_admin_api.py`。
    *   使用 `httpx` (一个现代的HTTP客户端)，编写**第一个会失败的集成测试**。
    *   **测试用例 (示例):**
        ```python
        # tests/api/test_admin_api.py
        import pytest
        from httpx import AsyncClient
        from main import app # 您的FastAPI应用实例
        
        @pytest.mark.asyncio
        async def test_get_arbitration_cases_unauthorized():
            async with AsyncClient(app=app, base_url="http://test") as ac:
                response = await ac.get("/api/v1/admin/arbitration-cases")
            # 断言：在没有认证的情况下，应该返回401或403
            assert response.status_code == 403
        ```

2.  **【绿灯 & 重构】用 `FastAPI` 实现API:**
    *   在 `packages/backend-python/src/api/` 目录下，创建`admin_router.py`。
    *   **使用Pydantic进行"契约定义":**
        *   用`Pydantic`的`BaseModel`，来**1:1地重新定义**您之前在TypeScript中创建的所有`DTO`和实体接口。**这是连接"类型安全"的关键一步。**
        *   **示例:**
            ```python
            # src/schemas/reports.py
            from pydantic import BaseModel, Field
            from datetime import date
            
            class GeneratedReport(BaseModel):
                report_id: int
                report_type: str
                target_code: str | None = None
                report_date: date
                # ...
            ```
    *   **创建FastAPI路由:**
        *   编写最简单的FastAPI路径操作函数，让测试通过。
        *   FastAPI会自动处理请求体验证（基于Pydantic模型）和响应格式化，并自动生成Swagger UI文档。

3.  **重复循环:**
    *   为`知识库CRUD`、`仲裁案件查询`、`提交反馈`等**每一个API端点**，重复"红-绿-重构"的循环，直到所有API功能都被完整地重建和测试覆盖。

## 第三阶段："业务逻辑"的无缝平移 (2-3天)

**目标:** 将之前分散在NestJS Service和Python脚本中的业务逻辑，整合到统一的Python Service层。

### 1. 创建Service层

*   在 `packages/backend-python/src/services/` 目录下，创建一系列的Service类。
*   例如：`arbitration_service.py`, `config_service.py`, `llm_gateway_service.py`。

### 2. 逻辑平移

*   将`_archive_backend_nestjs/src/services/`目录下的TypeScript业务逻辑，**翻译**成Python代码，并放入对应的Python Service类中。
*   **这是一个相对直接的工作，** 因为核心的业务流程（例如，如何计算"分歧度"）是与语言无关的。

### 3. 整合`main_workflow.py`的逻辑

*   将`main_workflow.py`、`qwen_analyzer.py`、`doubao_analyzer.py`等脚本中的**核心业务逻辑**（而不是流程控制），也**重构**并封装到相应的Service类中。
*   **例如:** `qwen_analyzer.py`中的所有分析步骤，都应该成为`QwenFactAnalyzerService`类的方法。

## 第四阶段：重构"主工作流"与最终联调 (1-2天)

**目标:** 用一个更优雅、更符合现代Python实践的方式，来重写`main_workflow.py`。

### 1. 引入`Typer`或`Click`

*   将`main_workflow.py`从一个简单的脚本，重构为一个**专业的命令行应用 (CLI)**。
*   `Typer`库能让您非常轻松地创建带参数、子命令的CLI。
*   **新命令示例:**
        ```bash
        python main.py run-daily-flow --date 2025-09-26
        python main.py run-historical-backfill --start-date 20230101 --end-date 20231231
        ```

### 2. 重构主工作流

*   `main.py`现在不再包含具体的业务逻辑。它只负责**解析命令行参数**和**调用Service层**。
*   **示例:**
        ```python
        # main.py
        import typer
        from services.workflow_service import WorkflowService
        
        app = typer.Typer()
        
        @app.command()
        def run_daily_flow(date: str):
            workflow_service = WorkflowService()
            workflow_service.run_daily_flow(date)

        if __name__ == "__main__":
            app()
        ```

### 3. 最终联调

*   运行`python main.py run-daily-flow ...`，确保整个从"数据获取"到"分析报告生成"的后端流程，在纯Python环境中，能够端到端地、正确无误地运行。
