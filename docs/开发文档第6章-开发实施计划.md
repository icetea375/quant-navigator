## 6.2 开发实施计划概述

**这是一个将TDD思想融入开发步骤的具体计划。**

### 6.2.1 开发原则

1. **TDD驱动**: 所有开发都必须遵循"红-绿-重构"循环
2. **测试优先**: 先写测试，再写代码
3. **质量保证**: 任何没有对应测试用例的生产代码提交都将被视为不合格
4. **持续集成**: GitHub Actions是最终的、不可逾越的"守门人"

### 6.2.2 开发阶段划分

- **Sprint 1**: 基础设施与"契约"定义 (预计1周)
- **Sprint 2**: 核心模块与工作流开发 (预计2周)
- **Sprint 3**: AI治理中心开发 (预计2周)
- **Sprint 4**: 知识库填充与系统联调 (预计1周)

## 6.3 Sprint 1: 基础设施与"契约"定义 (预计1周)

### 6.3.1 目标
建立TDD开发基础设施，定义所有核心"契约"（接口和数据结构）。

### 6.3.2 核心任务

#### 6.3.2.1 数据库实体TDD开发
**任务描述**: 为所有数据库实体编写单元测试，然后实现实体代码。

**具体步骤**:
1. **【写测试】** 编写所有数据库实体 (`entity.ts`) 的**单元测试**。
   - `tests/unit/entities/financial-reports.entity.test.ts`
   - `tests/unit/entities/generated-reports.entity.test.ts`
   - `tests/unit/entities/human-feedback-loop.entity.test.ts`
   - `tests/unit/entities/processed-events.entity.test.ts`
   - `tests/unit/entities/quant-signals.entity.test.ts`

2. **【写代码】** 编写所有实体代码，让测试通过。
   - 确保所有可为空字段都使用 `string | null` 联合类型
   - 实现所有 `class-validator` 装饰器
   - 确保实体关系正确映射

3. **【验证】** 运行所有实体测试，确保100%通过。

#### 6.3.2.2 核心接口TDD开发
**任务描述**: 为核心模块定义接口契约，然后实现基础功能。

**具体步骤**:
1. **【写测试】** 编写核心接口的集成测试骨架。
   - `tests/integration/api/admin.spec.ts` (可以先`skip`)
   - `tests/integration/api/llm-gateway.spec.ts`
   - `tests/integration/api/data-pipeline.spec.ts`

2. **【写代码】** 实现核心接口的基础功能。
   - 定义所有核心接口和类型
   - 实现基础的CRUD操作
   - 确保接口类型安全

#### 6.3.2.3 TDD基础设施搭建
**任务描述**: 搭建完整的TDD开发环境。

**具体步骤**:
1. **【配置测试环境】**
   - 配置Jest测试框架
   - 配置pytest测试框架
   - 配置测试数据库
   - 配置Mock服务

2. **【配置CI/CD流水线】**
   - 设置GitHub Actions
   - 配置自动化测试
   - 配置代码覆盖率检查
   - 配置代码质量检查

3. **【配置开发工具】**
   - 配置ESLint和Prettier
   - 配置TypeScript严格模式
   - 配置Git hooks
   - 配置IDE设置

### 6.3.3 验收标准
- [ ] 所有数据库实体测试通过
- [ ] 所有核心接口测试通过
- [ ] CI/CD流水线正常运行
- [ ] 代码覆盖率 > 90%
- [ ] 无TypeScript错误
- [ ] 无ESLint警告
