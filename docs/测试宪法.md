

### **【v1t0.11 测试宪法】—— 智能分析系统测试实施与验收指南**

**前言：** 本指南是"量化导航仪"项目所有测试开发的最高行为准则。所有提交的测试代码，无论由人类还是AI编写，都必须100%遵循此指南。任何违反本指南原则的代码提交，都将被视为**不合格**并被拒绝合并。

**核心哲学："一个人的军队"原则**

本项目遵循"一个人的军队"开发哲学，将**简单性**和**可维护性**作为最高优先级。我们拒绝"过度工程化"，追求"足够好"的解决方案。测试策略必须与这一哲学保持一致：避免引入不必要的复杂性，优先选择简单、直接、易于理解和维护的测试方法。

#### **第一章：测试的核心哲学——“测试即契约”**

**第1条：测试的唯一目的**
测试的唯一目的，是**验证“生产代码”是否严格履行了其“设计契约”**。测试代码本身不应包含任何“变通”或“魔法”。

**第2条：禁止“为了通过而测试”**
严禁任何为了让测试流程“变绿”而采取的“耍滑头”行为。**一个失败的测试，是宝贵的“风险信号”，而不是一个需要被“消灭”的麻烦。**

**第3条："红灯-绿灯-重构"原则**
所有开发都应遵循此原则：
1.  **红灯 (Red):** 先为尚未实现的功能，编写一个**会失败**的测试。
2.  **绿灯 (Green):** 编写最简单的生产代码，让这个测试**恰好通过**。
3.  **重构 (Refactor):** 在测试保持通过的前提下，优化和重构生产代码。

**第4条："简单性优先"铁律**
*   **拒绝过度工程化：** 严禁为了追求理论上的"完美"而引入不必要的复杂性。测试方案必须与项目的"一个人的军队"哲学保持一致。
*   **选择"足够好"的解决方案：** 当面临多种测试策略选择时，优先选择最简单、最直接、最容易理解和维护的方案。
*   **成本效益原则：** 测试方案的成本（包括学习成本、维护成本、认知负担）必须与其带来的收益相匹配。避免"用大炮打蚊子"。
*   **人工仲裁优于复杂自动化：** 当自动化解决方案变得过于复杂时，优先考虑简单的人工仲裁机制。

#### **第二章：代码级实施铁律**

**第5条：【类型安全铁律】严禁使用任何形式的"类型欺骗"**
为根除“空指针”和“类型不匹配”等地狱，在**所有 `.test.ts` / `.spec.ts` 文件**中：
*   **绝对禁止**使用 `as any`。
*   **绝对禁止**使用 `@ts-ignore` 或 `@ts-nocheck`。
*   **如果测试代码出现类型错误，** 唯一的、100%的解决方案是：**回到源头（`entity.ts`, `service.ts`等生产代码），去修复那个导致类型不匹配的、错误的类型定义。**

**第6条：【模拟（Mock）铁律】只模拟"外部边界"，不模拟"内部逻辑"**
*   **允许模拟的对象:**
    *   **外部API调用:** 数据库（Prisma/TypeORM客户端）、商业LLM API客户端、Tushare API客户端等。
    *   **Node.js/浏览器原生模块:** `fs`, `Date`等。
*   **绝对禁止模拟的对象:**
    *   **被测试模块的内部方法或其直接依赖的、同属于本系统的其他内部模块。** 如果你需要模拟一个内部方法才能让测试通过，这说明你的模块设计存在“高耦合”的坏味道。

**第7条：【断言（Assertion）铁律】断言必须"精确且有意义"**
*   **禁止"存在性"断言:** 避免使用 `expect(result).not.toBeNull()` 或 `expect(result).toBeDefined()` 这种几乎没有信息量的断言。
*   **必须使用"值"断言:** 断言必须检查**具体的、预期的**值。
    *   **好的实践:** `expect(result.status_code).toBe(200);`
    *   **好的实践:** `expect(result.data.length).toBeGreaterThan(0);`
    *   **好的实践:** `expect(result.error_message).toContain('Invalid credentials');`

**第8条：【统一入口铁律】前端测试的"创世函数"**
*   **本质定义:** `createTestWrapper`不是"模板"，而是**"工厂(Factory)"**、**"环境构造器(Environment Constructor)"**、**"创世函数(Genesis Function)"**。
*   **强制力体现:**
    *   **环境同构强制:** 强制执行与生产环境高度相似的"模拟驾驶舱"，包含完整Pinia、Vue Router、Element Plus。
    *   **状态隔离强制:** 通过`setActivePinia(createPinia())`强制每次测试拥有全新的、干净的"中央仪表盘"。
*   **封装性体现:**
    *   **复杂性封装:** 将Element Plus组件测试的复杂性（如`el-select`无限递归问题）完美封装在内部。
    *   **接口简化:** 普通测试编写者只需调用`createTestWrapper`，无需了解内部实现细节。
*   **宪法地位:** 本项目`frontend-main`包中，所有Vue组件的挂载，**必须**且**只能**通过调用位于`/src/utils/test-utils.ts`中的`createTestWrapper`函数来完成。任何直接调用`@vue/test-utils`的`mount`函数的行为，都将被视为严重的违宪行为。

#### **第三章：前端测试专门宪法——"真实浏览器战场"**

**前言：前端测试的终极战场**

本章节专门针对`packages/frontend-main`包的前端测试，建立更加严格、更加专业、更加不可妥协的测试标准。前端测试不是后端测试的简单移植，而是面向真实用户、真实浏览器、真实交互的"终极战场"。

**第9条：【测试的唯一目的——面向用户的行为契约】**
测试的唯一目的，是验证"生产代码"是否严格履行了其**"面向用户的、可观察的行为契约"**。测试**不应该**关心、也**不应该**断言任何关于代码**"内部实现细节"**的东西。

*   **合宪的行为：** `expect(screen.getByText('加载中...')).toBeInTheDocument();` (测试用户看到了什么)
*   **违宪的行为：** `expect(wrapper.vm.isLoading).toBe(true);` (测试内部状态，严重违宪)

**第10条：【"红灯-绿灯-重构"是唯一的开发节奏】**
所有新功能的开发、所有Bug的修复，都**必须**遵循T-D-D（测试驱动开发）的"红-绿-重构"循环。**禁止**编写任何一行没有先行为其编写一个"失败的测试"的生产代码。

**第11条：【禁止"为了通过而欺骗"】**
任何为了让测试"变绿"而采取的、削弱测试有效性的"欺骗"行为，都将被视为最高级别的叛国罪。这包括但不限于：
*   在生产代码中，添加`if (process.env.NODE_ENV === 'test')`之类的逻辑分支。
*   将被测试的行为本身注释掉。
*   将一个有意义的断言，修改为一个无意义的、恒为真的断言。

**第12条：【类型安全是绝对的】**
在所有`.test.ts`文件中：
*   **绝对禁止**使用`as any`。
*   **绝对禁止**使用`@ts-ignore`或`@ts-nocheck`。
*   **如果出现类型错误，唯一的、100%的解决方案是：** 回到源头（生产代码、类型定义、测试数据），去修复那个导致类型不匹配的、**错误的根源**。

**第13条：【模拟的边界是神圣的】**
*   **只允许模拟"跨越网络边界"的外部依赖。** 在本项目中，这**只包括**：
    1.  我们的"API Service层"（例如`arbitrationService.ts`）。
    2.  最底层的浏览器API（例如`window.matchMedia`）。
*   **绝对禁止模拟：**
    1.  Pinia Store本身。
    2.  Vue Router实例。
    3.  任何Element Plus组件。
    4.  任何我们自己编写的、属于本项目一部分的子组件或组合式函数。
*   **原则：** 我们的测试，必须在一个**尽可能接近"真实"**的、包含了**完整插件生态**的环境中运行。

**第14条：【断言必须面向"用户"】**
*   **优先使用`@testing-library/vue`的`screen`查询。** 我们必须以一个**真实用户（包括屏幕阅读器用户）**的视角，来查找和断言元素。
*   **查询的优先级顺序（不可违背）：**
    1.  `getByRole`: 按"可访问性"角色查找（最健壮）。
    2.  `getByLabelText`: 按表单标签查找。
    3.  `getByText`: 按元素显示的文本内容查找。
    4.  `getByTestId`: 作为**最后的、万不得已的**手段，为那些没有文本、没有角色的元素，添加`data-testid`属性。
*   **绝对禁止**使用基于CSS类名或HTML标签的、脆弱的选择器（例如`wrapper.find('.my-class > div')`）。

**第15条：【`createTestWrapper`是唯一的创世函数】**
本项目`frontend-main`包中，所有Vue组件的挂载，**【必须】**且**【只能】**通过调用位于`/src/utils/test-utils.ts`中的`createTestWrapper`函数来完成。任何直接调用`@vue/test-utils`的`mount`或`@testing-library/vue`的`render`的行为，都将被视为严重的违宪行为。

**第16条：【真实浏览器是唯一的战场】**
*   所有组件和集成测试，都**必须**运行在由`@vitest/browser`所提供的**真实浏览器环境**中（当前为Chromium）。
*   **彻底废除**并**禁止**使用任何基于`JSDOM`或`happy-dom`的模拟DOM环境。我们接受真实浏览器环境带来的"慢"，以换取其无可替代的"真"。

**第17条：【CI/CD是最终的、唯一的、无情的审判官】**
*   一个功能分支，只有在**同时满足**以下所有由CI/CD流水线**自动强制执行**的条件时，才**允许**被合并到主分支：
    1.  `pnpm run lint` **0个错误**。
    2.  `pnpm run type-check` **0个错误**。
    3.  `pnpm run test:coverage` **100%测试用例通过**。
    4.  由`vitest`生成的覆盖率报告，其**行覆盖率 (Lines) 不得低于85%**，**分支覆盖率 (Branches) 不得低于80%**。
*   **任何试图绕过、禁用或降低这些门禁标准**的行为，都将被视为对本项目根基的攻击。

#### **第四章：测试金字塔与分层策略**

##### **4.0 测试金字塔：从"模拟"到"真实"的信任阶梯**

我们遵循测试金字塔原则，在三个层次上逐步引入"真实性"，构建从"完全模拟"到"完全真实"的信任阶梯：

```
      / \
     / ▲ \  <-- 端到端测试 (E2E Tests) - 最高真实性，最少数量
    /_____\
   / ▲ ▲ \ <--- 集成测试 (Integration Tests) - 中等真实性，中等数量
  /_______\
 / ▲ ▲ ▲ \ <---- 单元测试 (Unit Tests) - 最低真实性，最多数量
/___________\
```

**第一层：单元测试 (Unit Tests) - 我们当前90%的工作**

*   **目的：** 在**绝对隔离、绝对可控**的环境下，验证一个**最小的、独立的"单元"**（一个函数、一个类）的内部逻辑是否正确。
*   **如何实现"隔离"：** 大量使用**模拟 (Mocking)**。我们用"假的"数据库连接、"假的"API客户端、"假的"文件系统，来替换掉所有它与之交互的"真实世界"。
*   **回答的问题：** "`calculate_value_score`这个函数，在给定PE=10, PB=1.5时，**是否**能精确地计算出我们期望的分数？"
*   **何时测试"真的"：** **永远不会。** 单元测试的哲学基础，就是**"不相信真实世界"**。因为真实世界是缓慢的、不稳定的、充满副作用的。单元测试追求的是**速度**和**确定性**。

**第二层：集成测试 (Integration Tests) - 我们下一步即将进入的领域**

*   **目的：** 验证**多个"单元"组合在一起时**，是否能协同工作。它开始引入有限的、受控的"真实性"。
*   **如何实现"有限真实"：** 我们会开始移除一部分模拟，换上**真实的、但用于测试的**组件。
    *   **例子1（我们为`get_mda_data`写的测试）：** 我们**没有**模拟数据库本身，我们连接了一个**真实的**（可能是内存中的SQLite，或是一个专用的Docker容器化的PostgreSQL）测试数据库。但我们**模拟了**写入这条数据的过程（在测试开始时手动插入）。
    *   **例子2（即将为`DataPipelineService`写的测试）：** 我们会**模拟**`TushareFetcher`（因为我们不想在每次测试时都真的去请求Tushare的API），但我们会让它与一个**真实的**测试数据库进行交互，以验证数据能否被正确写入。
*   **回答的问题：**
    - "`DataPipelineService`**是否**能将从`TushareFetcher`（的模拟对象）那里获取的数据，正确地写入到一个**真实的数据库**中？"
    - "`MainWorkflow`**是否**能正确地调用`DataPipelineService`和`QuantSignalService`（的模拟对象），并将结果正确地传递给`DatabaseManager`（的模拟对象）？"
*   **何时测试"真的"：** 在**模块与模块的"接缝"处**进行真实性测试。我们相信每个单元内部是好的（由单元测试保证），我们现在关心的是，它们"焊接"在一起的地方是否牢固。

**第三层：端到端测试 (E2E Tests) - 金字塔的顶端，最终的真相**

*   **目的：** **尽可能少地使用或完全不使用模拟**，在一个与生产环境**无限接近**的完整系统中，模拟一个**完整的用户（或系统）工作流**，以验证整个系统作为一个整体，是否能完成其核心使命。
*   **如何实现"最高真实"：**
    1.  我们会有一个**完整的、独立的、为E2E测试而存在的环境**。
    2.  这个环境包含了一个**真实的、被数据管道填充了近期真实数据的数据库**。
    3.  这个环境会运行**所有真实的后端服务**。
    4.  **唯一的模拟可能发生在最最外部的边界**：例如，我们可能会有一个"Tushare API的录制回放服务器"，或者一个"LLM API的模拟服务器"，它们会返回我们预先录制好的、真实的API响应。这是为了保证E2E测试的**可重复性**。
*   **回答的问题：** "当`main_workflow.py`的`run_daily_flow`被触发时，它**是否真的**能从一个包含真实市场数据的数据库中检测出异常，**是否真的**能生成两份报告，**是否真的**能将一个仲裁案件存入数据库，并且这个案件**是否真的**能被前端通过API正确地查询并展示出来？"
*   **何时测试"真的"：** **在最后。** E2E测试是昂贵的、缓慢的、脆弱的。它们是我们最后的、也是最有力的信心来源。我们只会为我们系统**最核心、最关键的1-3条"黄金路径"**编写E2E测试。

##### **4.1 单元测试 (Unit Tests) - 验收标准**

*   **验收标准:**
    1.  每一个独立的**公共方法 (public method)**，都必须有对应的单元测试。
    2.  每一个**数据校验规则 (`class-validator`)**，都必须有至少一个**成功**和一个**失败**的测试用例。
    3.  代码覆盖率 (Code Coverage) 的**门禁 (Gateway):** **行覆盖率不得低于85%，分支覆盖率不得低于80%。** 任何低于此标准的提交将被自动拒绝。

##### **4.2 集成测试 (Integration Tests) - 验收标准**

*   **验收标准:**
    1.  每一个**API端点 (Endpoint)**，都必须有集成测试。
    2.  测试用例必须覆盖**"成功路径 (Happy Path)"**、**"认证/授权失败"**和**"输入数据校验失败"**这三种核心场景。
    3.  对于涉及数据库写入的测试，**必须**在测试结束后，通过 `afterEach` 或 `afterAll` 钩子，**清理掉**本次测试创建的所有数据，确保测试的独立性和可重复性。

##### **4.3 端到端测试 (E2E Tests) - 有限真实性原则**

*   **验收标准:**
    1.  必须覆盖**至少一条**最核心的、贯穿整个系统的用户工作流。对于我们，就是 **"异常事件的归因流程"** 和 **"AI治理中心的仲裁流程"**。
    2.  **E2E测试的"有限真实性"原则：**
        *   **准生产环境：** E2E测试运行在与生产环境**几乎完全一样**的独立环境中，包含真实的测试数据库和真实的API密钥。
        *   **真实API调用：** E2E测试**将真实地、通过互联网**调用Tushare和LLM的API，但使用**固定的、几乎不会变化的**输入数据。
        *   **接受不确定性：** 我们承认E2E测试**偶尔会因为网络或外部API的抖动而失败**，并**接受**这种不完美。
        *   **人工仲裁机制：** E2E测试在CI/CD中设置为**"非阻塞 (Non-blocking)"**模式，失败时发送告警，由人类专家判断是"真实世界的暂时性抖动"还是"真正的代码bug"。
    3.  **避免脆弱的选择器:** 严禁使用 `div:nth-child(3)` 这种脆弱的CSS选择器。测试代码必须使用**专门为测试设计的、稳定的属性选择器**，例如 `cy.get('[data-testid="submit-feedback-button"]')`。

#### **第五章：自动化执行与强制约束**

**第18条：所有测试必须在提交前通过**
*   在 `package.json` 的 `scripts` 中，配置一个 `pre-commit` 或 `pre-push` 的 **Git钩子 (Git Hook)**。
*   **指令:** `npm run test`。
*   **效果:** 任何开发者（包括AI）在尝试提交代码时，本地的所有测试都会被**自动触发**。只要有一个测试失败，**`git commit` 或 `git push` 命令就会被自动阻止**。

**第19条：CI/CD流水线是最终的"守门人"**
*   在您的代码托管平台（GitHub Actions, GitLab CI）上，配置一个**持续集成 (Continuous Integration)** 流水线。
*   **流水线步骤:**
    1.  `Checkout Code`
    2.  `Install Dependencies`
    3.  `Run Lint & Format Check`
    4.  **`Run All Tests (Unit + Integration)`**
    5.  **`Check Code Coverage`**
*   **规则:** 只有当**所有步骤都100%通过**时，一个Pull Request才**允许被合并**到主分支



**第20条：【环境一致性铁律】所有测试必须运行在一个与生产环境"同构"的测试环境中**

*   **虚拟环境激活:** 对于Python项目，**必须**在运行测试前激活正确的虚拟环境。本项目使用**根目录的单一虚拟环境**：`source venv/bin/activate` (Linux/macOS) 或 `venv\Scripts\activate` (Windows)。
*   **依赖一致:** 测试环境**必须**通过 `pnpm install`，安装与 `package.json` 中**完全一致**的生产和开发依赖。
*   **框架一致:** 对于后端NestJS测试，**必须**使用 `@nestjs/testing` 模块提供的 `Test.createTestingModule()` 来构建测试模块。
*   **绝对禁止的行为:**
    *   **禁止**为了"简化测试"而手动实例化服务类（`new MyService()`），必须通过NestJS的依赖注入容器来获取实例。
    *   **禁止**修改或简化 `test-setup.ts` 和 `test-teardown.ts` 中的核心环境搭建逻辑。

**第21条：【CI/CD作为最终"环境仲裁者"】**

*   **最终裁决:** 无论一个测试在本地是如何被"耍滑头"地运行通过的，它最终都必须通过**在CI/CD流水线中，那个干净的、标准的、从零开始构建的Docker环境中**的测试。
*   **强化CI/CD配置:**
    *   CI脚本必须**严格执行** `npm install` -> `npm run test` 的流程。
    *   CI流水线是**不可被开发者（或AI）修改**的"最终考场"，它确保了测试环境的**绝对纯洁性**。

#### **第六章：配置管理准则**

**第22条：【配置统一管理铁律】**
*   **配置分层原则：** 系统配置必须严格遵循三层架构：环境配置层 (`config/env/`) > 服务配置层 (`packages/backend-python/config/`) > 数据库配置层 (`system_configs` 表)。
*   **单一事实来源：** 所有配置必须通过 `config/settings.py` 统一管理，**禁止**在代码中硬编码配置值。
*   **配置继承：** 子模块的测试配置必须继承根目录的配置，**禁止**重复定义。

**第23条：【测试环境隔离】**
*   **测试配置独立：** 测试环境必须使用独立的配置文件 (`config/env/env.test`)，**禁止**使用生产或开发环境配置。
*   **容器隔离：** 所有测试必须运行在Docker容器化的独立环境中，**禁止**使用本地开发环境进行测试。
*   **数据隔离：** 每个测试用例执行前必须清理测试数据，确保测试间的**完全隔离**。

#### **第七章：测试基础设施管理**

**第24条：【测试执行统一化】**
*   **唯一执行入口：** 所有测试必须通过项目根目录的 `run-tests.sh` 脚本执行，**禁止**直接调用测试框架命令。
*   **Monorepo依赖管理：** 测试依赖必须通过PNPM统一管理，**禁止**在子目录中独立安装依赖。
*   **测试目录结构：** 测试文件必须按照 `tools/tests/` 目录结构组织，**禁止**随意放置测试文件。

**第25条：【测试数据管理】**
*   **种子数据标准：** 所有测试数据必须通过 `libs/database/seeds/` 目录下的标准化脚本初始化。
*   **Mock数据规范：** 外部API调用必须使用统一的Mock数据，存储在 `tools/tests/data/mocks/` 目录。
*   **测试Fixtures：** 可重用的测试数据必须定义为Fixtures，存储在 `tools/tests/data/fixtures/` 目录。

**第26条：【测试工具统一化】**
*   **框架版本统一：** 所有测试框架版本必须在根目录的 `package.json` 中统一管理。
*   **工具函数共享：** 测试辅助函数必须在 `tools/tests/utils/` 中定义，供所有测试使用。
*   **测试报告统一：** 所有测试结果必须输出到 `tools/tests/reports/` 目录，使用统一的报告格式。

#### **第八章：版本控制与工作流**

**前言：** 本章节旨在将《测试宪法》的原则，从代码的编写层面，延伸到代码的**生命周期管理**层面。Git的每一次操作，都应被视为执行宪法的一部分。

**第27条：【原子化提交铁律】一次提交只做一件事，且必须完整**

*   **定义：** "原子化提交"指的是，每一次 `git commit` 都必须代表一个**单一的、逻辑上完整的**工作单元。
*   **TDD与提交的绑定：** 一个完整的"红灯-绿灯-重构"循环，在功能实现后，通常就对应着一次原子化提交。
*   **提交的完整性：**
    *   **生产代码和测试代码必须在一个提交中。** 它们是同一枚硬币的两面，绝对禁止分离。
    *   **禁止**在一个提交中混入不相关的修改（例如，在实现新功能的同时，顺手修复了另一个模块的拼写错误）。
*   **实践准则：** 在执行 `git commit` 前，先使用 `git status` 和 `git diff --staged` 检查暂存区，确保只有与本次逻辑单元相关的代码被提交。

**第28条：【提交信息即文档铁律】提交信息必须清晰、规范、可追溯**

*   **规范：** 所有提交信息**必须**遵循"**[Conventional Commits](https://www.conventionalcommits.org/)**"规范。这不仅是为了好看，更是为了让机器（例如自动生成更新日志的工具）和人类都能轻松理解每一次变更的意图。
*   **格式：** `<类型>(<范围>): <主题>`
    *   **类型 (Type):**
        *   `feat`: 新功能
        *   `fix`: 生产环境的Bug修复
        *   `refactor`: 不改变外部行为的代码重构
        *   **`test`: 增加或修复测试**
        *   `docs`: 文档修改
        *   `chore`: 构建流程、辅助工具等变更
    *   **范围 (Scope):** 可选，指明本次提交影响的模块。例如 `feat(DataPipeline):`
    *   **主题 (Subject):** 简明扼要地描述变更内容。
*   **好的实践示例：**
    ```
    test(QuantSignal): 增加对异常处理分支的覆盖

    为 calculate_quant_signal, detect_anomalies,
    save_quant_signal, get_quant_signal_by_id
    这四个方法的通用异常处理块，添加了单元测试。

    使用 `pytest.raises` 来断言当依赖项失败时，
    是否正确抛出了预期的自定义业务异常。
    ```

**第29条：【主干常绿，分支工作铁律】主分支永远是可部署的**

*   **主干保护：** `main` (或 `master`) 分支是神圣的。任何时候，`main` 分支的代码都必须是**100%通过所有测试**的、随时可以部署到生产环境的状态。
*   **特性分支工作流：**
    1.  所有新的开发工作（无论是新功能还是bug修复），都**必须**从最新的 `main` 分支创建一条新的**特性分支**（例如 `feat/add-user-auth` 或 `fix/data-pipeline-nan-error`）。
    2.  所有的开发和**原子化提交**都在这条特性分支上进行。
    3.  当特性分支的工作完成后，**必须**通过一个**拉取请求 (Pull Request / Merge Request)** 来申请合并回 `main` 分支。
*   **合并的唯一前提：** 一个Pull Request**只允许**在通过了第四章第9条所规定的、**所有CI/CD流水线检查**（包括Lint、测试、覆盖率门禁）之后，才能被合并。

#### **第九章：文档即基石 - 对抗"知识熵增定律"**

**前言：** 在本项目中，文档不是事后的"写作任务"，而是开发过程中不可分割的一部分。代码解释"如何实现"，测试验证"做了什么"，而文档则阐明**"为何如此"**。所有文档都必须与代码一样，接受严格的版本控制和质量管理。

**第30条：【文档的"单一事实来源"铁律】**

*   本项目所有的人类可读的、持久化的技术文档，其**唯一的、排他的**事实来源，**必须**是位于项目根目录下的 `/docs` 目录。
*   文档格式**必须**为 Markdown (`.md`)。
*   **绝对禁止**任何形式的"自建文档"，包括但不限于本地的Word/Pages文档、备忘录、离线的个人笔记等。所有知识，如果要被沉淀，就必须进入这个受版本控制的 `/docs` 目录。
*   **文档分类标准：**
    *   **架构文档**: `docs/开发文档第X章-*.md` - 系统设计、技术架构、模块关系
    *   **API文档**: `docs/API_*.md` - 接口规范、参数说明、响应格式
    *   **配置文档**: `docs/配置*.md` - 环境配置、部署指南、运维手册
    *   **测试文档**: `docs/测试*.md` - 测试策略、测试用例、质量保证
    *   **变更文档**: `docs/*_CHANGELOG.md` - 版本变更、功能更新、破坏性变更

**第31条：【代码的"自文档"铁律】**

*   代码本身是第一层级的文档。所有**公开的 (public)** 模块、类、函数和方法，都**必须**拥有符合 **Google Python Style Guide** 规范的**文档字符串 (Docstring)**。
*   **Docstring的核心任务**是解释：
    1.  这个单元的**意图和职责**是什么？
    2.  它的**参数 (`Args`)** 分别代表什么？
    3.  它的**返回值 (`Returns`)** 是什么？
    4.  它可能会**抛出 (`Raises`)** 哪些我们自定义的业务异常？
    5.  它的**使用示例 (`Examples`)** 是什么？
*   **注释的职责：** 普通的行内注释（`#`）**不应该**解释代码"正在做什么"（代码本身应该足够清晰），而应该解释**"为什么用这种方式做"**，特别是当代码包含复杂的算法、业务权衡或为了绕过某个特定问题而采取的"非常规"实现时。
*   **文档字符串示例：**
    ```python
    def calculate_quant_signal(
        self,
        market_data: pd.DataFrame,
        threshold: float = 0.05
    ) -> QuantSignal:
        """
        计算量化信号，基于市场数据异常检测。

        该方法通过分析市场数据的统计特征，识别潜在的异常模式，
        并生成相应的量化交易信号。信号强度基于异常程度和置信度。

        Args:
            market_data: 市场数据DataFrame，必须包含'price'和'volume'列
            threshold: 异常检测阈值，默认0.05（5%）

        Returns:
            QuantSignal: 包含信号强度、置信度和元数据的信号对象

        Raises:
            DataValidationError: 当市场数据格式不正确时
            CalculationError: 当信号计算过程中出现数值错误时

        Examples:
            >>> signal = engine.calculate_quant_signal(df, 0.03)
            >>> print(f"Signal strength: {signal.strength}")
        """
    ```

**第32条：【文档与开发同步铁律】**

*   文档的更新，**不是**开发和测试完成后的一个独立步骤，而是它们的**一部分**。
*   **"完成的定义 (Definition of Done)"**：一个功能、修复或重构，只有在以下所有条件都满足时，才能被认为是"完成的"：
    1.  生产代码已实现。
    2.  单元/集成测试已通过。
    3.  **所有相关的Docstring和`/docs`目录下的架构文档，都已同步更新。**
    4.  **所有受影响的API文档和配置文档，都已同步更新。**
*   **Pull Request审查：** 代码审查 (Code Review) **必须**将"文档是否同步更新"作为一个核心的审查项。一个缺少必要文档更新的PR，应被视为**不完整的**，并被拒绝合并。
*   **文档审查清单：**
    *   [ ] 新增/修改的公共接口是否有完整的Docstring？
    *   [ ] 架构变更是否更新了相关设计文档？
    *   [ ] API变更是否更新了接口文档？
    *   [ ] 配置变更是否更新了部署文档？
    *   [ ] 测试策略变更是否更新了测试文档？

**第33条：【知识传承的"设计决策记录"铁律】**

*   所有重要的**设计决策**、**架构权衡**和**业务逻辑背景**，都**必须**被记录在 `docs/ADR.md` (Architecture Decision Records) 中。
*   **ADR格式标准：**
    ```markdown
    # ADR-001: 选择PostgreSQL作为主数据库

    ## 状态
    已接受

    ## 背景
    我们需要一个支持复杂查询和事务的数据库...

    ## 决策
    选择PostgreSQL作为主数据库...

    ## 后果
    正面：强大的SQL支持、ACID事务...
    负面：学习曲线、运维复杂度...
    ```
*   **必须记录的设计决策类型：**
    *   技术选型决策（数据库、框架、库的选择）
    *   架构模式决策（微服务 vs 单体、事件驱动 vs 请求响应）
    *   业务规则决策（异常检测算法、评分规则、阈值设定）
    *   性能优化决策（缓存策略、数据库索引、API设计）
    *   安全决策（认证方式、权限模型、数据加密）

**第34条：【文档质量的"可维护性"铁律】**

*   文档必须保持**可读性**、**准确性**和**时效性**。
*   **可读性标准：**
    *   使用清晰、简洁的语言
    *   采用一致的格式和结构
    *   提供适当的代码示例和图表
    *   避免技术术语的过度使用
*   **准确性保证：**
    *   文档内容必须与代码实现保持一致
    *   定期审查和更新过时信息
    *   验证代码示例的正确性
    *   确保链接的有效性
*   **时效性维护：**
    *   每次重大功能更新后，必须审查相关文档
    *   建立文档更新提醒机制
    *   定期进行文档质量审计

**第35条：【文档的"版本控制"铁律】**

*   所有文档都必须接受与代码相同的版本控制管理。
*   **提交规范：** 文档更新必须遵循与代码相同的提交信息规范：
    *   `docs(模块名): 更新API文档`
    *   `docs(架构): 添加ADR-002数据库分片决策`
    *   `docs(配置): 更新部署指南`
*   **变更追踪：** 重大文档变更必须通过Pull Request进行审查和合并。
*   **历史保留：** 文档的变更历史必须完整保留，以便追踪决策的演进过程。

---

**第八章总结：**

通过建立"文档即基石"的宪法原则，我们构建了项目的**三大支柱**：

1.  **代码 (Code):** 负责实现 - "如何做"
2.  **测试 (Tests):** 负责验证 - "做了什么"
3.  **文档 (Docs):** 负责解释 - "为何如此"

这三者共同构成了一个**健康的、可维护的、能够抵抗时间侵蚀的**项目生态系统，有效对抗"知识的熵增定律"，防止"项目失忆症"的发生。
