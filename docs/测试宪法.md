

### **【v1t0.10 测试宪法】—— 智能分析系统测试实施与验收指南**

**前言：** 本指南是“量化导航仪”项目所有测试开发的最高行为准则。所有提交的测试代码，无论由人类还是AI编写，都必须100%遵循此指南。任何违反本指南原则的代码提交，都将被视为**不合格**并被拒绝合并。

#### **第一章：测试的核心哲学——“测试即契约”**

**第1条：测试的唯一目的**
测试的唯一目的，是**验证“生产代码”是否严格履行了其“设计契约”**。测试代码本身不应包含任何“变通”或“魔法”。

**第2条：禁止“为了通过而测试”**
严禁任何为了让测试流程“变绿”而采取的“耍滑头”行为。**一个失败的测试，是宝贵的“风险信号”，而不是一个需要被“消灭”的麻烦。**

**第3-条：“红灯-绿灯-重构”原则**
所有开发都应遵循此原则：
1.  **红灯 (Red):** 先为尚未实现的功能，编写一个**会失败**的测试。
2.  **绿灯 (Green):** 编写最简单的生产代码，让这个测试**恰好通过**。
3.  **重构 (Refactor):** 在测试保持通过的前提下，优化和重构生产代码。

#### **第二章：代码级实施铁律**

**第4条：【类型安全铁律】严禁使用任何形式的“类型欺骗”**
为根除“空指针”和“类型不匹配”等地狱，在**所有 `.test.ts` / `.spec.ts` 文件**中：
*   **绝对禁止**使用 `as any`。
*   **绝对禁止**使用 `@ts-ignore` 或 `@ts-nocheck`。
*   **如果测试代码出现类型错误，** 唯一的、100%的解决方案是：**回到源头（`entity.ts`, `service.ts`等生产代码），去修复那个导致类型不匹配的、错误的类型定义。**

**第5条：【模拟（Mock）铁律】只模拟“外部边界”，不模拟“内部逻辑”**
*   **允许模拟的对象:**
    *   **外部API调用:** 数据库（Prisma/TypeORM客户端）、商业LLM API客户端、Tushare API客户端等。
    *   **Node.js/浏览器原生模块:** `fs`, `Date`等。
*   **绝对禁止模拟的对象:**
    *   **被测试模块的内部方法或其直接依赖的、同属于本系统的其他内部模块。** 如果你需要模拟一个内部方法才能让测试通过，这说明你的模块设计存在“高耦合”的坏味道。

**第6条：【断言（Assertion）铁律】断言必须“精确且有意义”**
*   **禁止“存在性”断言:** 避免使用 `expect(result).not.toBeNull()` 或 `expect(result).toBeDefined()` 这种几乎没有信息量的断言。
*   **必须使用“值”断言:** 断言必须检查**具体的、预期的**值。
    *   **好的实践:** `expect(result.status_code).toBe(200);`
    *   **好的实践:** `expect(result.data.length).toBeGreaterThan(0);`
    *   **好的实践:** `expect(result.error_message).toContain('Invalid credentials');`

#### **第三章：测试分类与验收标准**

##### **3.1 单元测试 (Unit Tests)**

*   **验收标准:**
    1.  每一个独立的**公共方法 (public method)**，都必须有对应的单元测试。
    2.  每一个**数据校验规则 (`class-validator`)**，都必须有至少一个**成功**和一个**失败**的测试用例。
    3.  代码覆盖率 (Code Coverage) 的**门禁 (Gateway):** **行覆盖率不得低于85%，分支覆盖率不得低于80%。** 任何低于此标准的提交将被自动拒绝。

##### **3.2 集成测试 (Integration Tests)**

*   **验收标准:**
    1.  每一个**API端点 (Endpoint)**，都必须有集成测试。
    2.  测试用例必须覆盖**“成功路径 (Happy Path)”**、**“认证/授权失败”**和**“输入数据校验失败”**这三种核心场景。
    3.  对于涉及数据库写入的测试，**必须**在测试结束后，通过 `afterEach` 或 `afterAll` 钩子，**清理掉**本次测试创建的所有数据，确保测试的独立性和可重复性。

##### **3.3 端到端测试 (E2E Tests)**

*   **验收标准:**
    1.  必须覆盖**至少一条**最核心的、贯穿整个系统的用户工作流。对于我们，就是 **“异常事件的归因流程”** 和 **“AI治理中心的仲裁流程”**。
    2.  **禁止在E2E测试中使用任何数据模拟 (Mock)。** E2E测试必须运行在一个与生产环境**完全一致**的、包含真实数据库和后端服务的环境中。
    3.  **避免脆弱的选择器:** 严禁使用 `div:nth-child(3)` 这种脆弱的CSS选择器。测试代码必须使用**专门为测试设计的、稳定的属性选择器**，例如 `cy.get('[data-testid="submit-feedback-button"]')`。

#### **第四章：自动化执行与强制约束**

**第7条：所有测试必须在提交前通过**
*   在 `package.json` 的 `scripts` 中，配置一个 `pre-commit` 或 `pre-push` 的 **Git钩子 (Git Hook)**。
*   **指令:** `npm run test`。
*   **效果:** 任何开发者（包括AI）在尝试提交代码时，本地的所有测试都会被**自动触发**。只要有一个测试失败，**`git commit` 或 `git push` 命令就会被自动阻止**。

**第8条：CI/CD流水线是最终的“守门人”**
*   在您的代码托管平台（GitHub Actions, GitLab CI）上，配置一个**持续集成 (Continuous Integration)** 流水线。
*   **流水线步骤:**
    1.  `Checkout Code`
    2.  `Install Dependencies`
    3.  `Run Lint & Format Check`
    4.  **`Run All Tests (Unit + Integration)`**
    5.  **`Check Code Coverage`**
*   **规则:** 只有当**所有步骤都100%通过**时，一个Pull Request才**允许被合并**到主分支



**第9条：【环境一致性铁律】所有测试必须运行在一个与生产环境“同构”的测试环境中**

*   **定义“同构”:**
    *   **依赖一致:** 测试环境**必须**通过 `npm install` (或 `pnpm install`)，安装与 `package.json` 中**完全一致**的生产和开发依赖。
    *   **框架一致:** 对于后端NestJS测试，**必须**使用 `@nestjs/testing` 模块提供的 `Test.createTestingModule()` 来构建一个**完整的、包含依赖注入容器的NestJS测试模块**。
    *   **配置一致:** 测试环境应加载与生产环境结构相同的配置文件，仅修改数据库连接等环境变量。
*   **绝对禁止的行为:**
    *   **禁止**为了“简化测试”而手动实例化服务类（`new MyService()`），必须通过NestJS的依赖注入容器来获取实例。
    *   **禁止**修改或简化 `test-setup.ts` 和 `test-teardown.ts` 中的核心环境搭建逻辑。

**第10条：【执行入口唯一性铁律】所有测试的执行，必须通过项目根目录的“唯一官方入口”**

*   **唯一入口:** 项目根目录 `package.json` 中定义的 `scripts` 命令，例如 `npm run test:unit`, `npm run test:integration`。
*   **绝对禁止的行为:**
    *   **禁止**进入子目录（如 `aigc/backend/`），单独运行 `jest` 或其他测试命令。
    *   **禁止**为测试创建临时的、非官方的启动脚本。
*   **理由:** 确保每一次测试，都在一个**统一的、预设的、包含了所有必要环境设置和依赖加载**的上下文中被执行。

**第11条：【CI/CD作为最终“环境仲裁者”】**

*   **最终裁决:** 无论一个测试在本地是如何被“耍滑头”地运行通过的，它最终都必须通过**在CI/CD流水线中，那个干净的、标准的、从零开始构建的Docker环境中**的测试。
*   **强化CI/CD配置:**
    *   CI脚本必须**严格执行** `npm install` -> `npm run test` 的流程。
    *   CI流水线是**不可被开发者（或AI）修改**的“最终考场”，它确保了测试环境的**绝对纯洁性**。
