## 0.2 TDD铁律 - 第0.1条

### 0.2.1 核心原则

**必须遵循"测试驱动开发 (Test-Driven Development)"**

- **核心原则**: **先写测试，再写代码。**
- **开发流程**: 所有新功能或Bug修复，都必须严格遵循**"红-绿-重构 (Red-Green-Refactor)"**的循环。

### 0.2.2 红-绿-重构循环详解

#### 0.2.2.1 红 (Red) - 红灯阶段
- **目标**: 先为目标功能，编写一个**会失败**的测试用例
- **作用**: 这个测试用例将**定义**出新功能的接口、输入和期望输出
- **要求**: 测试用例必须能够编译和运行，但会失败（因为功能还未实现）

#### 0.2.2.2 绿 (Green) - 绿灯阶段
- **目标**: 编写**最少量**的生产代码，恰好让这个测试用例**通过**
- **原则**: 只写能让测试通过的最简代码，不要过度设计
- **要求**: 测试必须通过，但代码可以很简陋

#### 0.2.2.3 重构 (Refactor) - 重构阶段
- **目标**: 在测试保持通过的前提下，优化和重构生产代码的内部实现
- **原则**: 改进代码质量，但功能行为不变
- **要求**: 所有测试必须继续通过

### 0.2.3 验收标准

**任何没有对应测试用例的生产代码提交，都将被视为**不合格**。**

## 0.3 TDD实施指南

### 0.3.1 测试类型分层

#### 0.3.1.1 单元测试 (Unit Tests)
- **范围**: 测试单个函数或类的行为
- **目标**: 验证最小可测试单元的正确性
- **工具**: Jest (JavaScript/TypeScript), pytest (Python)
- **要求**: 快速执行，无外部依赖

#### 0.3.1.2 集成测试 (Integration Tests)
- **范围**: 测试多个组件之间的交互
- **目标**: 验证组件间的接口和数据流
- **工具**: 与单元测试相同，但包含外部依赖
- **要求**: 使用测试数据库和模拟服务

#### 0.3.1.3 端到端测试 (E2E Tests)
- **范围**: 测试完整的用户工作流
- **目标**: 验证整个系统从用户角度的工作流程
- **工具**: Playwright (Web), Cypress (Web)
- **要求**: 模拟真实用户操作

### 0.3.2 测试编写原则

#### 0.3.2.1 AAA模式 (Arrange-Act-Assert)
```typescript
// Arrange - 准备测试数据
const input = { stockCode: '000001', date: '2025-01-17' };
const expectedOutput = { prediction: 0.75, confidence: 0.8 };

// Act - 执行被测试的功能
const result = await predictionEngine.generatePrediction(input);

// Assert - 验证结果
expect(result.prediction).toBeCloseTo(expectedOutput.prediction, 2);
expect(result.confidence).toBeGreaterThan(0.7);
```

#### 0.3.2.2 测试命名规范
- **格式**: `should [expected behavior] when [condition]`
- **示例**: `should return high confidence when stock has strong fundamentals`
- **原则**: 测试名称应该清楚地表达测试的意图

#### 0.3.2.3 测试独立性
- **原则**: 每个测试应该独立运行，不依赖其他测试
- **要求**: 测试之间不能有共享状态
- **实现**: 使用 `beforeEach` 和 `afterEach` 清理测试环境

### 0.3.3 测试覆盖率要求

#### 0.3.3.1 最低覆盖率标准
- **单元测试**: 90% 以上代码覆盖率
- **集成测试**: 80% 以上关键路径覆盖率
- **E2E测试**: 100% 核心用户场景覆盖率

#### 0.3.3.2 覆盖率监控
- **工具**: Jest Coverage, Istanbul (JavaScript), Coverage.py (Python)
- **要求**: CI/CD流水线中强制执行覆盖率检查
- **标准**: 低于最低覆盖率的代码不允许合并

## 0.4 开发流程规范

### 0.4.1 Git工作流

#### 0.4.1.1 分支策略
- **main**: 生产环境代码，必须通过所有测试
- **develop**: 开发环境代码，集成所有功能分支
- **feature/**: 功能开发分支，从develop创建
- **hotfix/**: 紧急修复分支，从main创建

#### 0.4.1.2 提交规范
```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**类型 (type)**:
- `feat`: 新功能
- `fix`: Bug修复
- `test`: 测试相关
- `refactor`: 重构
- `docs`: 文档更新

**示例**:
```
feat(prediction): add confidence scoring to prediction engine

- Implement confidence calculation based on historical accuracy
- Add confidence threshold filtering
- Update prediction API to include confidence score

Closes #123
```

### 0.4.2 代码审查流程

#### 0.4.2.1 Pull Request要求
- **测试**: 必须包含相应的测试用例
- **覆盖率**: 新代码必须达到覆盖率要求
- **文档**: 必须更新相关文档
- **审查**: 至少需要一位审查者批准

#### 0.4.2.2 审查检查清单
- [ ] 代码遵循TDD原则
- [ ] 测试用例覆盖所有新功能
- [ ] 测试用例通过
- [ ] 代码覆盖率达标
- [ ] 代码风格一致
- [ ] 文档已更新
- [ ] 无安全漏洞

### 0.4.3 CI/CD流水线

#### 0.4.3.1 自动化测试流程
1. **代码检查**: ESLint, Prettier, TypeScript编译
2. **单元测试**: 运行所有单元测试
3. **集成测试**: 运行集成测试套件
4. **E2E测试**: 运行端到端测试
5. **覆盖率检查**: 验证测试覆盖率
6. **安全扫描**: 依赖项安全检查

#### 0.4.3.2 部署流程
1. **测试环境**: 自动部署到测试环境
2. **验收测试**: 运行验收测试套件
3. **生产部署**: 手动触发生产环境部署
4. **监控验证**: 部署后监控系统健康状态

## 0.5 质量保证体系

### 0.5.1 代码质量工具

#### 0.5.1.1 静态分析
- **ESLint**: JavaScript/TypeScript代码检查
- **Prettier**: 代码格式化
- **SonarQube**: 代码质量分析
- **TypeScript**: 类型检查

#### 0.5.1.2 动态分析
- **Jest**: 测试框架和覆盖率
- **Playwright**: E2E测试
- **Lighthouse**: 性能分析
- **Security Audit**: 安全漏洞扫描

### 0.5.2 性能要求

#### 0.5.2.1 响应时间标准
- **API响应**: < 200ms (95%请求)
- **页面加载**: < 2s (首屏)
- **数据库查询**: < 100ms (简单查询)
- **LLM调用**: < 5s (单次调用)

#### 0.5.2.2 资源使用限制
- **内存使用**: < 512MB (单个服务)
- **CPU使用**: < 80% (平均)
- **磁盘IO**: < 100MB/s (读写)
- **网络带宽**: < 10Mbps (平均)

### 0.5.3 错误处理标准

#### 0.5.3.1 错误分类
- **系统错误**: 服务器内部错误，需要立即修复
- **业务错误**: 业务逻辑错误，需要记录和监控
- **用户错误**: 用户输入错误，需要友好提示
- **网络错误**: 外部服务错误，需要重试机制

#### 0.5.3.2 错误处理原则
- **快速失败**: 尽早发现和报告错误
- **优雅降级**: 部分功能失败时保持核心功能
- **详细日志**: 记录足够的错误信息用于调试
- **用户友好**: 向用户提供有意义的错误信息
