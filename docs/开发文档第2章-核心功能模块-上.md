### 2.0.1 核心工作流 (main_workflow.py - 单体串行脚本)

**技术栈**: Python + 商业LLM API
**职责**: 系统的"主干流程"

**核心架构**:
- **主干**: 整个在线流程由一个**单一的Python脚本 `main_workflow.py`** 驱动
- **分支**: 脚本内部逻辑高度模块化，通过**函数/类调用**的方式执行支持模块功能
- **智能**: 所有的"思考"工作（语义理解、逻辑推理、预测生成），**全部外包**给顶级的商业LLM API
- **容错**: 脚本中的每一个核心步骤，都必须用 `try...except` 块包裹，并记录详细日志

**工作流程**:
```python
def main_workflow():
    """主工作流 - 单体串行脚本"""
    try:
        # 1. 加载待分析股票列表
        stock_list = load_stock_list()
        
        # 2. 数据融合与特征工程
        features = data_fusion_and_feature_engineering(stock_list)
        
        # 3. 异常事件检测 (LightGBM)
        anomalies = detect_anomalies(features)
        
        # 4. 对每个异常股票执行分析
        for stock_code in anomalies:
            try:
                # 4.1 执行MD&A Verifier v2.1
                mda_result = mda_verifier.verify_fulfillment(stock_code)
                
                # 4.2 执行事件链构建
                event_chain = build_event_chain(stock_code, date)
                
                # 4.3 执行最终预测(多场景概率)
                prediction = generate_prediction(stock_code, date)
                
                # 4.4 执行反事实验证 (DoWhy)
                counterfactual = verify_counterfactual(stock_code, prediction)
                
                # 4.5 保存结果到数据库
                save_results(stock_code, mda_result, event_chain, prediction, counterfactual)
                
            except Exception as e:
                log_error(f"处理股票 {stock_code} 时出错: {e}")
                continue
                
    except Exception as e:
        log_error(f"主工作流执行失败: {e}")
```

```python
class LLMService:
    def __init__(self, api_key, base_url):
        self.api_key = api_key
        self.base_url = base_url
    
    def call_llm(self, prompt, max_tokens=2000):
        """调用商业LLM API"""
        try:
            response = requests.post(
                f"{self.base_url}/v1/chat/completions",
                headers={"Authorization": f"Bearer {self.api_key}"},
                json={
                    "model": "gpt-4",
                    "messages": [{"role": "user", "content": prompt}],
                    "max_tokens": max_tokens
                }
            )
            return response.json()["choices"][0]["message"]["content"]
        except Exception as e:
            log_error(f"LLM API调用失败: {e}")
            return None
```

### 2.1.1 MD&A Verifier v2.1 (模块化实现)

**职责**: 验证管理层履约记录

**实现**:
- 作为一个Python类 (`MDAVerifier`)，包含 `verify_fulfillment`, `verify_forecast`, `assess_quality` 等方法
- 所有需要语义理解的地方（如提取"承诺"、判断"兑现"），都通过**调用商业LLM API**完成
- **关键**: 引入"智能仲裁"的预审逻辑，只有"AI内部分歧大"或"风险高"的案例，才会被打上 `'HUMAN_REVIEW_REQUIRED'` 标签

**核心方法**:
```python
class MDAVerifier:
    def __init__(self, llm_service):
        self.llm_service = llm_service
    
    def verify_fulfillment(self, stock_code, date):
        """验证管理层履约记录"""
        try:
            # 1. 获取MD&A文本
            mda_text = self.get_mda_text(stock_code, date)
            
            # 2. 构建Prompt
            prompt = self.build_verification_prompt(mda_text)
            
            # 3. 调用LLM API
            result = self.llm_service.call_llm(prompt)
            
            # 4. 解析结果
            verification_result = self.parse_verification_result(result)
            
            # 5. 判断是否需要人工仲裁
            if self.needs_human_review(verification_result):
                verification_result['status'] = 'HUMAN_REVIEW_REQUIRED'
            
            return verification_result
            
        except Exception as e:
            log_error(f"MD&A验证失败: {e}")
            return None
```

### 2.1.4 反事实验证 (模块化实现)

**职责**: 基于DoWhy的因果验证

**实现**:
- 一个Python函数 `verify_counterfactual(stock_code, prediction)`
- 使用DoWhy库进行因果推理和反事实分析
- 结合LLM API进行语义理解和解释

**核心实现**:
```python
def verify_counterfactual(stock_code, prediction):
    """反事实验证"""
    try:
        # 1. 构建因果图
        causal_graph = build_causal_graph(stock_code)
        
        # 2. 使用DoWhy进行因果推理
        causal_model = CausalModel(
            data=get_stock_data(stock_code),
            treatment='anomaly_event',
            outcome='price_change',
            graph=causal_graph
        )
        
        # 3. 估计因果效应
        causal_effect = causal_model.estimate_effect(
            method_name="backdoor.linear_regression"
        )
        
        # 4. 反事实分析
        counterfactual = causal_model.do(x=1)  # 假设事件发生
        
        # 5. 使用LLM解释结果
        explanation_prompt = f"""
        基于以下因果分析结果，解释反事实验证：
        
        因果效应: {causal_effect}
        反事实结果: {counterfactual}
        原始预测: {prediction}
        
        请分析这些结果的一致性和可信度。
        """
        
        explanation = llm_service.call_llm(explanation_prompt)
        
        return {
            'causal_effect': causal_effect,
            'counterfactual': counterfactual,
            'explanation': explanation,
            'consistency_score': calculate_consistency(prediction, counterfactual)
        }
        
    except Exception as e:
        log_error(f"反事实验证失败: {e}")
        return None
```

### 2.2.3 智能仲裁流程

```python
def needs_human_review(verification_result):
    """判断是否需要人工仲裁"""
    try:
        # 1. 检查AI内部分歧
        if verification_result.get('ai_disagreement_score', 0) > 0.7:
            return True
        
        # 2. 检查风险等级
        if verification_result.get('risk_level', 'LOW') in ['HIGH', 'CRITICAL']:
            return True
        
        # 3. 检查置信度
        if verification_result.get('confidence_score', 1.0) < 0.6:
            return True
        
        # 4. 检查历史判例相似度
        if get_similar_case_count(verification_result) == 0:
            return True
        
        return False
        
    except Exception as e:
        log_error(f"判断是否需要人工仲裁失败: {e}")
        return True  # 出错时默认需要人工仲裁
```

### 2.3.3 核心API接口

**主要接口**: `GET /api/v1/admin/arbitration-cases/:caseId`

**请求参数**:
- `caseId`: 案件ID (格式: `{stock_code}_{date}`)
- `include_panels`: 可选，指定需要的数据面板 (默认全部)

**响应格式**:
```typescript
interface ArbitrationCaseData {
  case_id: string;
  stock_code: string;
  date: string;
  panels: {
    raw_text: RawTextData;

    quant_signals: QuantSignals;

```

**实现**:
```typescript
@Get('arbitration-cases/:caseId')
async getArbitrationCaseData(
  @Param('caseId') caseId: string,
  @Query('include_panels') includePanels?: string[]
): Promise<ArbitrationCaseData> {
  const [stockCode, date] = caseId.split('_');
  const panels = includePanels || ['raw_text', 'financial_snapshot', 'quant_signals', 'flow_and_chips', 'historical_arbitrations'];
  
  // 并行获取所有面板数据
  const panelPromises = [];
  
  if (panels.includes('raw_text')) {
    panelPromises.push(this.getRawTextData(caseId));
  }
  if (panels.includes('financial_snapshot')) {
    panelPromises.push(this.getFinancialSnapshot(stockCode));
  }
  if (panels.includes('quant_signals')) {
    panelPromises.push(this.getQuantSignals(stockCode, date));
  }
  if (panels.includes('flow_and_chips')) {
    panelPromises.push(this.getFlowAndChipsData(stockCode, date));
  }
  if (panels.includes('historical_arbitrations')) {
    panelPromises.push(this.getHistoricalArbitrations(stockCode, await this.getIndustry(stockCode)));
  }
  
  const panelResults = await Promise.all(panelPromises);
  
  return {
    case_id: caseId,
    stock_code: stockCode,
    date: date,
    panels: this.organizePanelData(panels, panelResults),
    metadata: {
      generated_at: new Date().toISOString(),
      data_freshness: await this.getDataFreshness(caseId),
      panel_count: panels.length
    }
  };
}
```

### 2.4.4 Prompt模板库 🆕 v10.1 新增

**核心设计原则**:
- **角色明确**: 清晰定义LLM应该扮演的专家角色
- **上下文清晰**: 提供所有必要的、无歧义的背景信息和数据

**四大核心模板库**:

1. **MD&A验证Prompt模板** (`mda_verification_prompts.json`):
   - 角色: 极其审慎、只相信白纸黑字、且擅长财务和商业逻辑的审计师
   - 功能: 提取经营承诺、验证履约情况、计算管理层可信度

2. **事件链构建Prompt模板** (`event_chain_building_prompts.json`):
   - 角色: 熟悉商业逻辑、对时间线极其敏感的历史学家
   - 功能: 串联孤立事件、分析因果关系、验证时间线一致性
   - 关键陷阱: "后视镜"偏见，必须严格排除股价数据

3. **预测生成Prompt模板** (`prediction_generation_prompts.json`):

4. **反事实验证Prompt模板** (`counterfactual_validation_prompts.json`):
   - 角色: 专门"抬杠"的、持怀疑态度的"魔鬼代言人"
   - 功能: 挑战高置信度预测、设计压力测试、分析认知偏差
   - 关键陷阱: "为了反对而反对"，需要将风险作为降权项而非直接否定

### 2.1.2 预测引擎 (Prediction Engine) - 全新增模块

#### 2.1.2.1 精确角色与定位

`Prediction Engine` (PE) 是系统的 **"首席策略师"** 和 **"Alpha发现引擎"**。它与"归因引擎"并行存在，但目标截然不同：

- **核心目标**: **预测未来 (Predict the Future)**。通过对3年历史数据的深度学习，为市场上的每一个标的，输出一个关于其未来短期收益的**概率性预测**。
- **上游 (Upstream)**: 它的数据输入完全来自于数据库中由 `DataPipeline` 和 `QuantSignalEngine` 准备好的**结构化特征**（量化信号、财务因子）和**非结构化文本**（新闻、公告）。
- **下游 (Downstream)**: 它的唯一输出是每日的**预测信号**，存储于数据库的 `daily_predictions` 表中。这些信号将作为：
  1. `Attribution Engine` (归因引擎) 判断**"市场预期"**的核心基线。
  2. 未来交易策略回测与实盘应用的基础。

#### 2.1.2.2 具体的技术实现方案

该引擎采用**分层、异步**的技术方案，在 **Python** 环境中实现。

**技术栈**:
- **语言**: Python 3.10+
- **核心库**: `Pandas` (数据处理), `LightGBM` (基础层模型), `Hugging Face Transformers` (增强层模型), `PyTorch` / `TensorFlow` (深度学习框架), `SQLAlchemy` (数据库交互)
- **服务框架**: `FastAPI` (用于包装成内部API), `Celery` (用于异步执行训练和预测任务)

**执行流程 (日常在线)**:
1. **触发**: `App_Service_Layer` 中的 `Orchestrator` 在每日 `17:00` 触发一个 Celery 异步任务，启动预测流程
2. **数据加载**: 预测脚本从数据库中加载当日最新的特征数据
3. **模型加载**: 从 `Model Registry` (模型注册表，可以是本地文件目录或云存储) 加载经过离线训练的**三层模型文件**
4. **分层预测**:
   a. 结构化特征被送入**基础层模型 (LightGBM)**，生成初步预测 `P1`
   b. 文本数据被送入**增强层模型 (FinBERT)**，生成情绪预测 `P2`
   c. `P1`, `P2` 及关键元数据被送入**决策层模型 (Fine-tuned LLM)**，生成最终预测 `P_final`
5. **结果存储**: 将 `P1`, `P2`, `P_final` 写入数据库的 `daily_predictions` 表

#### 2.1.2.3 三层模型的具体配置和参数

##### 基础层 (Layer 1): The Tabular Data Expert
- **模型选型**: **LightGBM (梯度提升树)**
  - **理由**: 对结构化表格数据处理速度快、性能强大、可解释性相对较好，是业界的成熟方案
- **核心配置与参数**:
  - `objective`: `'regression_l1'` (L1损失，对异常值更鲁棒)
  - `metric`: `'mse'` (均方误差)
  - `n_estimators`: `1000` (树的数量)
  - `learning_rate`: `0.05`
  - `num_leaves`: `31`
  - `feature_fraction`: `0.8` (特征采样)
  - `bagging_fraction`: `0.8` (数据采样)
  - `bagging_freq`: `1`

##### 增强层 (Layer 2): The Unstructured Information Expert
- **模型选型**: **FinBERT (领域预训练的BERT模型)**
  - **理由**: 专门在金融文本上进行过预训练，能更好地理解新闻、公告中的语义和情感
- **核心配置与参数**:
  - **任务**: 文本分类 (情感分析)
  - **输出**: `{-1, 0, 1}` 分别代表 **负面、中性、正面** 的情感倾向得分 `P2`
  - **微调**: 使用公开的中文金融情感数据集或内部标注数据，对 FinBERT 模型进行微调，使其更适应A股语境
  - **输入**: 将当日与某股票相关的所有新闻、公告标题拼接成一个长文本作为输入

##### 决策层 (Layer 3): The Chief Investment Officer
- **模型选型**: **Llama 3 8B (经过指令微调)**
  - **理由**: 具备强大的逻辑推理能力，且模型规模适中，适合进行领域知识的微调
- **核心配置**:
  - **任务**: **指令微调 (Instruction Fine-tuning)**
  - **目标**: 学习如何根据基础层和增强层的**"专家意见"**，结合上下文，做出一个更精准的最终回归预测
  - **输入**: 一个包含结构化信息的 Prompt (详见第5节)
