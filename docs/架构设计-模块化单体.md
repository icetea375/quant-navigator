# 架构设计 - 模块化单体

## 架构概述

本项目采用**模块化单体 (Modular Monolith)** 架构模式，这是一种在单体应用内部实现模块化设计的架构风格。

## 架构定义

### 什么是模块化单体？

**模块化单体**是一种架构模式，具有以下特征：

1. **单一部署单元**：整个应用作为一个单元进行部署
2. **内部模块化**：应用内部按业务功能划分为独立的模块
3. **明确接口**：模块间通过明确定义的接口进行通信
4. **独立开发**：每个模块可以独立开发、测试和维护
5. **未来可扩展**：模块可以轻松拆分为独立的微服务

### 与微服务的区别

| 特征 | 模块化单体 | 微服务 |
|------|------------|--------|
| 部署 | 单一部署单元 | 独立部署 |
| 数据存储 | 共享数据库 | 独立数据库 |
| 通信 | 进程内调用 | 网络调用 |
| 复杂度 | 相对简单 | 复杂 |
| 性能 | 高性能 | 网络延迟 |
| 一致性 | 强一致性 | 最终一致性 |

## 项目架构设计

### 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                    模块化单体应用                            │
├─────────────────────────────────────────────────────────────┤
│  Frontend Layer (Vue.js + React)                           │
│  ├── 主前端应用 (packages/frontend-main)                    │

│  └── 工作流模块 (workflow)                                 │
├─────────────────────────────────────────────────────────────┤
│  Business Logic Layer (Python)                             │

```

### 模块划分

#### 1. 仲裁模块 (Arbitration Module)
- **职责**：处理仲裁案例的创建、处理和跟踪
- **接口**：REST API + 内部服务接口
- **数据**：仲裁案例表、相关配置表

#### 2. 报告模块 (Reports Module)
- **职责**：生成和管理各种分析报告
- **接口**：REST API + 内部服务接口
- **数据**：报告表、模板表

#### 3. 工作流模块 (Workflow Module)
- **职责**：协调各个模块的工作流程
- **接口**：内部服务接口
- **数据**：工作流状态表、任务表

#### 4. LLM模块 (LLM Module)
- **职责**：提供AI能力支持
- **接口**：内部服务接口
- **数据**：LLM配置、对话历史

### 模块间通信

#### 1. 同步通信
- **REST API**：前端与后端模块间
- **函数调用**：后端模块间
- **事件发布/订阅**：模块间解耦通信

#### 2. 数据共享
- **共享数据库**：所有模块共享PostgreSQL数据库
- **缓存层**：Redis提供跨模块缓存
- **文件存储**：共享文件系统存储

### 技术栈

#### 前端层
- **Vue.js 3**：主前端应用
- **React 18**：AIGC前端应用
- **TypeScript**：类型安全
- **Element Plus / Ant Design**：UI组件库

#### 后端层
- **FastAPI**：API框架
- **Python 3.11+**：主要编程语言
- **Pydantic**：数据验证
- **SQLAlchemy**：ORM

#### 数据层
- **PostgreSQL 15**：主数据库
- **Redis 6**：缓存和会话存储
- **Alembic**：数据库迁移

#### 基础设施
- **Docker**：容器化
- **pnpm**：包管理
- **Jest/Vitest**：测试框架
- **ESLint/Prettier**：代码质量

## 优势

### 1. 开发效率
- **简单部署**：单一部署单元，减少运维复杂度
- **快速开发**：模块间直接调用，无网络延迟
- **统一工具链**：共享开发工具和流程

### 2. 性能优势
- **高性能**：进程内调用，无网络开销
- **强一致性**：共享数据库，ACID事务
- **资源效率**：共享资源，减少资源消耗

### 3. 维护性
- **模块化设计**：清晰的模块边界
- **独立开发**：团队可以并行开发不同模块
- **易于测试**：模块可以独立测试

### 4. 未来扩展性
- **渐进式拆分**：可以逐步拆分为微服务
- **技术栈统一**：便于技术栈升级
- **团队协作**：减少跨团队协调成本

## 实施原则

### 1. 模块边界清晰
- 每个模块有明确的职责范围
- 模块间通过接口通信，不直接访问内部实现
- 数据访问通过服务层，不直接访问数据库

### 2. 接口设计优先
- 先定义模块接口，再实现具体功能
- 接口设计考虑未来扩展性
- 使用版本控制管理接口变更

### 3. 数据一致性
- 使用数据库事务保证数据一致性
- 避免跨模块的数据直接访问
- 通过事件机制处理跨模块的数据同步

### 4. 测试策略
- 每个模块有独立的单元测试
- 模块间有集成测试
- 端到端测试覆盖关键业务流程

## 未来演进

### 阶段1：模块化单体（当前）
- 单一部署单元
- 内部模块化设计
- 共享数据库

### 阶段2：模块化单体+事件驱动
- 引入事件总线
- 模块间异步通信
- 提高系统解耦度

### 阶段3：微服务化（可选）
- 将模块拆分为独立服务
- 独立部署和扩展
- 分布式架构

## 总结

模块化单体架构为项目提供了：

1. **简单性**：相比微服务，减少了运维复杂度
2. **性能**：进程内调用，高性能
3. **一致性**：强一致性，易于保证数据正确性
4. **可维护性**：模块化设计，便于维护和扩展
5. **未来性**：为未来的微服务化留下可能性

这种架构模式特别适合：
- 中小型团队
- 业务逻辑相对集中的应用
- 对性能要求较高的场景
- 需要快速迭代的项目
